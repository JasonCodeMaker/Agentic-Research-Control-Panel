Dashboard Role

This page is the project overview surface. It gives the user a clear picture of how research is organized, and renders the operating context from its owners — the Scope SSOT objective, the schema.js route enum, the rules registry, and the project profile — before any package is opened. The dashboard stores none of that content itself.

The dashboard does not own claims, gates, validity, or evidence anchors. Those live on package surfaces under packages/<research-id>/, governed by the trust rules in trustworthy-research-rules.html.

Read first — Learnings

Open learnings.html before proposing a new direction, refinement, or experiment idea. It is the project-wide index of structured methodsTried rows for every adopted win, archived failure, and abandoned brainstorm — grouped by which paper-spine contribution they touch.

Trigger conditions: (i) before idea-discovery or research-refine; (ii) before converting a brainstorm idea into a package; (iii) whenever the user asks "what have we tried?", "why did X fail?", or "did we already do this?". Reading it first prevents re-deriving a method that already has a recorded verdict.

Research packages

Each package is one card. Filter by lane or by next route, sort by recency, and open a package to see its plan, runs, and results.

HTML Rules (R1-R18)

Form rules: how every HTML file under research_html/ is shaped. Cover semantic structure, single-URL sharing, stable data-* anchors, web-of-files linking, and forbidden patterns such as ASCII visuals and remote scripts.

Trust Rules (T1-T24)

Content rules: what package surfaces must say. Cover claim gates, validity classes, evidence anchors, last-updated freshness, and stage-specific page contracts. These rules apply to package pages, not to the dashboard chrome.
