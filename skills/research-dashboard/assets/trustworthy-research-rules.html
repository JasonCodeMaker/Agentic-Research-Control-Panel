trust rules - binding

Trustworthy Research Rules

Rules that the HTML surface must satisfy so a package-run agent's outputs stay grounded, scoped, and audit-ready for the user. Every rule below is binding for files under research_html/.

Dashboard README HTML Rules scope: research_html/

Scope & Authority

Authority order, highest first: invocation, project rules, WORKFLOW.md for state, gate, and route definitions, html-rules.html for HTML form, this file for content trust, then individual package content. html-rules.html defines how the HTML is shaped; this file defines what it must say. Rules in this file carry the prefix T so they do not collide with the R rules in html-rules.html.

A

User / Agent Contract

T1

No consequential transition without a user ack

The agent must not move a package from brainstorm to in-progress, advance into EXPERIMENT_RUNNING, record a result with verdict pass, move a package into the success lane, or move a package into the fail or archive lane without a user ack recorded on the surface for that specific transition.

T2

First viewport answers six items

Every package page and module page must display, inside the first viewport: workflow state, active gate, primary metric versus that gate, last decision and its evidence path, next route, and the current blocker if one exists. On a page where any of these items is not yet defined (such as a brainstorm page without a metric or gate) the field renders as unmeasured. Detail beyond these six items must be linked, not embedded inline.

T3

Recalled content is unverified evidence

Anything the agent recalls from a prior session, prior package, or training data is unverified. The agent must revalidate it against the live filesystem, runtime artifact, or external source before it can support a transition or appear as a measured value on the surface.

B

Hallucination Rules

T4

Citations and verification claims are resolved structured fields

Every cited file path, URL, paper, function, dataset, or method named on the surface, and every claim that the agent has inspected, opened, or checked an artifact, must appear inside an element carrying a structured field attribute (such as data-field) and must resolve to an existing path together with the timestamp of inspection. A citation or verification claim that cannot be resolved renders as unmeasured.

T5

Numbers carry an artifact path and timestamp, otherwise unmeasured

Every numeric value on the surface (metric, latency, count, percentage, rank, recall) must carry, at the same element, the artifact path that produced it and the timestamp at which that artifact was last modified. A numeric or status field with no resolved artifact path renders as the literal value unmeasured. Hedge phrases (appears, looks like, seems, trending, approximately), estimates, recalls, and interpolated guesses are not allowed in fields that hold measured values, gate verdicts, or transition reasons; such fields are either a measured value or unmeasured.

T6

Verified and recalled render as distinct surface states

Verified content (revalidated in the current session per T3) and recalled content (carried from prior sessions or training data) must be visually and attributively distinct on the surface and must never share a single representation.

T7

Only primary artifacts back transition decisions

Logs, csv files, checkpoints, and metric summary files generated by the run are primary artifacts. Tracker entries, prior agent narratives, and result-prose paragraphs are not primary artifacts. A transition decision must cite a primary artifact.

C

Misalignment Rules

T8

Hypothesis is restated at every transition page

Every page that owns a transition restates the active plan's hypothesis. The restatement must be string-equal to the plan's hypothesis field after lowercasing and whitespace normalization. A mismatch is treated as a misalignment and must be reconciled before the transition.

T9

Every measurement carries metric, dataset, and protocol

Every measurement on the surface carries, at the same element: the metric's name, formula, inputs, deduplication rule, and cutoff; the dataset, split, version or query-set hash, and pool scope; and the evaluation protocol's setting, candidate budget, beam or top-K, reranker, and seed. A result whose protocol does not match the active plan gate is recorded with verdict diagnostic_only and is never recorded with verdict pass. A measurement missing any of these fields is rejected as evidence.

T10

Counts of results are reported per validity class

Counts of results are reported per validity class (valid, diagnostic_only, failed, missing) and are never aggregated across these classes.

T11

Baseline reference metadata travels with every comparison

Every comparison to a baseline names, at the comparison element, the baseline's source, checkpoint, protocol, and last-verified date.

T12

Artifact pointers carry their owning package id

Every artifact pointer on a package page, including baseline pointers, carries the id of the package that owns the pointed-to artifact. A pointer whose owner does not match the page is rejected as evidence.

T13

Checkpoint and code commit are pinned at the metric

Every metric produced by a run is paired, at the metric element, with the exact checkpoint path and the exact git commit of the code that produced it. The pair appears next to the metric, not in a separate footer or document.

T14

Performance-affecting algorithm changes are surfaced as discrete cards

Any change to a model component (architecture, loss, training schedule, sampling, decoding, indexing, or evaluation logic) that may affect performance is surfaced as a discrete change card naming the component, the code anchor in file:function form, the one-sentence description of the change, the expected sign in {+, -, 0} and magnitude band in {small, medium, large}, and the experiments validating it.

T15

Living experiments expose live-status fields

Every experiment in a non-terminal status (queued, running, paused, stale, completed-pending-record) appears on the package surface with: state, last-log timestamp as the liveness indicator, retry count with prior failure entries listed, ETA, runtime root, and the recommended live action together with the cited plan threshold when an early stop is recommended.

T16

No-change boundary is restated and affirmed at every transition

Each transition page restates the active no-change boundary, lists the owned-files set declared at the start of the change, and carries an explicit affirmation that the boundary was not crossed. An edit outside the owned-files set is surfaced as a boundary violation.

T17

Core docs are kept up to date with every state-changing action

PLAN.md, TRACKER.md, RESULTS.md, and the package's HTML pages are updated in the same session as any state-changing action (implementation, launch, live check, result recording, or route change). Each page displays its last-update timestamp and is marked with data-stale="true" on the surface when its timestamp predates the latest state-changing action recorded for the package.

D

Stage Surface Rules

T18

Brainstorm pages display

The direction in one sentence; a flag stating whether the direction preserves or changes the project contribution spine defined in project rules; resolved citations to prior work; a flag if any prior package with category fail shares the same direction.

T19

Plan pages display

The hypothesis as a falsifiable claim; the primary metric; the baseline; the budget gate; the seed plan; a one-sentence diff against the prior plan; the no-change boundary.

T20

Implementation pages display

The owned-files set; a diff summary of changed files; an explicit no-change affirmation for the no-change boundary; each reviewer's verdict; the consolidated main-agent decision.

T21

Launch readiness displays (on tracker.html)

The Launch readiness card on tracker.html displays the resolved GPU id, CUDA_VISIBLE_DEVICES, conda environment, git commit, and dataset path, each marked with verified status; the expected runtime; the dry-run or smoke-test status. This card is the single home for pre-launch readiness (the prior launch.html page has been folded into tracker.html).

T22

Live monitoring displays (on tracker.html)

The per-run cards section on tracker.html displays, per open experiment: a rendered curve of the objective over time (when meaningful); the timestamp of the most recent log line; the count of missed scheduled checks; the recommended next live action with the cited plan threshold. The 10-minute cadence ledger lives in the latest live check table further down the same page. Tracker is the single home for live state (the prior live.html page has been folded into tracker.html).

T23

Result pages display

A result gate row with all of validity, baseline, plan gate, observed metric, budget, seed, artifacts, verdict, and reason; a supported-claims card; an unsupported-claims card; the protocol-match verdict; the artifact path with its modification timestamp.

T24

Next-action pages display

The chosen route from the allowed set per the WORKFLOW.md route table; every route considered and rejected, each with a one-sentence reason; the one-sentence reason for the chosen route, citing the evidence path it relies on.

Binding scope: every HTML file under research_html/. Update this file when a rule needs to change. Companion file: html-rules.html defines the form rules; this file defines the content-trust rules.
