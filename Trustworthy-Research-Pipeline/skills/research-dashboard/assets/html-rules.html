refinement rules - binding

HTML Rules

These rules apply to every refinement made under research_html/ from this point forward. They are derived from Thariq's article "The Unreasonable Effectiveness of HTML" (May 8, 2026) and the constructive replies in that thread.

Dashboard README Trust Rules scope: research_html/

Scope & Authority

These rules govern files inside research_html/ only. The canonical Markdown surfaces under research/active/<id>/{PLAN,TRACKER,RESULTS}.md remain the source of truth for plans, tracker rows, and result entries; the HTML surface presents and extends them. When a rule below conflicts with the active WORKFLOW.md or research_html/README.md, the lower-numbered rule here wins for HTML-surface decisions; WORKFLOW.md still wins for workflow state, gates, routes, and stop conditions.

A

Required Benefits to Realize

Each refinement must visibly deliver at least one of these benefits. If a change does not move toward any of them, do not make the change.

R1 - Long-form readability

Stay legible past 100 lines

Pages must be skimmable past the 100-line MD threshold. Use sections, sticky navigation, tabs, expand/collapse, and mobile-responsive layout so a reader can locate the relevant card without scrolling the whole document.

Applies to: dashboard, package home, module pages.

R2 - Rich representation

Use HTML's native expressiveness

Use tables, CSS-styled design data, SVG illustrations, embedded code, images, and inline canvas/spatial layout where they reduce ambiguity. Never approximate rich content with ASCII diagrams, unicode-color characters, or text-only stand-ins for figures.

Applies to: every module that displays metrics, comparisons, or structure.

Land plots and diagrams in-page

R3 - Visualizations inline

Training curves, latency-recall scatter plots, ablation grids, architecture diagrams, candidate-set heatmaps, and similar artifacts must be embedded inline (SVG preferred, PNG/JPG when unavoidable) on the package or module page they support, not linked away as external attachments.

Applies to: Results, Plan figures, Docs evidence cards.

R4 - Single-URL shareable

Self-contained, one-link sharing

Every page must render correctly when opened directly from disk or served as a static file. No build step, no dynamic server, no authenticated CDN. External assets are limited to images explicitly committed under the package or shared assets/ folder.

Applies to: every HTML file under research_html/.

R5 - Two-way interactivity

Make HITL controls a first-class element

Where a human decision drives the next agent step, expose a control: a filter, sort, comparison toggle, prompt-builder form, approve/ack/veto button, or copyable paste-back token. The page must be a HITL surface, not a status report.

Applies to: Tracker live-check rows, Results comparisons, decision gates.

R6 - Stable agent anchors

Every editable unit has a stable id

Every card, field, table row, decision, artifact path, command, and gate carries a stable data-* or id attribute the agent can target without parsing layout. New modules must extend the existing taxonomy (data-card, data-module, data-field, data-artifact, data-decision, data-table, data-exp-id, data-route, data-gate, data-workflow-state) before inventing a new anchor name.

Applies to: every renderer in assets/research.js and every template.

R7 - Edit-by-prompt friendly

Optimize for "Claude edits, human reads"

Assume edits arrive via prompted patches, not hand-typing. Predictable card structure, one logical concept per card, stable anchors, and explicit field names beat compact prose. A reviewer should still be able to follow a diff, but human typing is no longer the dominant authoring path for HTML files.

Applies to: card layout decisions, field naming, JSON-in-HTML schemas.

R8 - Web of files, not one mega page

Decompose into linked HTML pages

Keep the dashboard - category - package - module - docs decomposition. New investigative artifacts (failure analyses, candidate previews, ablation explorations) become their own HTML page under packages/<id>/docs/, linked from the owning module. Do not pile new content into one growing page.

Applies to: package growth, docs additions, new investigations.

R9 - Stream-safe structure

Render usefully even when partial

Structure pages so that each top-level section is independently useful. Do not put critical state (workflow state, current gate, next route) below large content blocks. The first viewport must always answer "what is this and what is next."

Applies to: package home, module pages.

R18 - Audience-tagged context

Collapse agent-only context by default

Content that is agent-important only - continuity notes, internal reasoning, file-map pointers, "remember for next turn" - must be wrapped in <details data-audience="agent"><summary>agent context</summary>...</details> so the browser collapses it for human readers while it stays in the source for agents. Both-audience and human-only content renders inline without <details>. The data-audience="agent" attribute extends the R6 stable-anchor taxonomy. The chat-output form of this rule wraps agent-only blocks in a markdown > blockquote.

Applies to: every package page that carries agent continuity notes; dashboard chrome that exposes agent context.

B

Guardrails (must respect)

R10 - Markdown stays canonical for plans

Do not delete or bypass MD source files

PLAN.md, TRACKER.md, and RESULTS.md in research/active/<id>/ remain the canonical, PR-reviewable, hand-or-agent-edited surface. HTML pages render or extend them; they do not replace them. Any sync script must be lossless from MD plus runtime JSON to HTML, in that direction by default.

Applies to: any sync, render, or migration script.

R11 - Semantic HTML, not visual bloat

Prefer machine-readable structure over decoration

Use <section>, <article>, <table>, <details>, <form>, <script type="application/json"> and data-* attributes. Avoid CSS-only animations, decorative canvas, drag-and-drop fluff, and any visual bling that does not help an agent or a reader extract a fact.

Applies to: every new template, every CSS addition.

R12 - PR-reviewable changes

Keep diffs reviewable

HTML is harder to review than MD. Mitigate: one logical card per edit when possible; do not reflow entire files; do not minify; keep one element per line where it aids readability; commit generated and hand-edited HTML in separate commits when both change.

Applies to: every commit touching research_html/.

R13 - Don't over-systematize

Skills scaffold, content stays free-form

The existing research-dashboard and research-package skills define the shared chrome and required overview cards. They must not be extended to demand a fixed schema for every investigative HTML file inside packages/<id>/docs/. One-off HTML investigations stay free-form and are linked from the Docs module.

Applies to: skill definitions, package contracts, module library.

C

Forbidden Patterns

R14 - No ASCII or unicode visual hacks

No text-art stand-ins

ASCII tables of metrics that should be a real <table>, unicode bar charts, unicode color squares, and ASCII flow diagrams are not acceptable inside research_html/. Use HTML tables, inline SVG, or committed images instead.

R15 - No build-step dependencies

No bundlers, no frameworks

No React, Vue, webpack, Vite, npm, or any toolchain that requires a build step before pages render. Keep assets/research.css and assets/research.js as plain static files. Embed small JSON via <script type="application/json">; embed small JS in assets/ only when needed.

R16 - No silent claim aggregation

Diagnostic vs valid must stay visible

Never aggregate valid, diagnostic_only, failed, or missing rows into a single number without a visual or attribute-level distinction. The user's MSR-VTT Setting 2 compact-budget rule, the Stage-1 vs Stage-2 distinction, and seed/baseline/budget mismatches must stay visible in the UI, not collapsed into prose.

R17 - No dynamic external dependencies

No remote scripts, fonts from disk only

Pages must not load remote JavaScript, remote analytics, remote CSS, or external CDN fonts. Everything required to render lives under research_html/assets/ or under the owning package directory. This protects offline review and keeps the single-URL share model honest.

D

How to Apply These Rules

Per-refinement checklist

  1. State which benefit (R1-R9, R18) the change realizes. If none, do not make the change.
  2. Confirm guardrails R10-R13 are not violated.
  3. Confirm no forbidden pattern (R14-R17) is introduced.
  4. Reuse an existing data-* anchor before inventing a new one.
  5. Render the page from disk and from the dashboard before declaring the change done.
  6. Commit hand-edits and generated HTML in separate commits if both change.

When MD is still the right answer

Plans, trackers, results, and any artifact that lives in PR review remain in research/active/<id>/*.md. The HTML surface renders and extends them, adds visualizations, exposes HITL controls, and presents runtime state. It does not replace the MD.
