Is the Business Task-Management Paradigm the Right Foundation for Human↔AI Alignment?

A source-grounded evaluation of whether Jira / Linear / Asana-style project management — and the CSCW theory beneath it — should anchor the bidirectional context-alignment layer ("Solve #2") of an autonomous idea→paper research pipeline.

Jira / Linear / Asana Agile · Kanban · OKR Articulation Work (Schmidt & Bannon) Boundary Objects (Star & Griesemer) Common Ground (Clark & Brennan) Shared Mental Models Human–AI Teaming

Abstract / Verdict

VERDICT: HYBRID — adopt the mechanisms, reject the throughput metaphor

The question is whether the business task & project-management paradigm is an optimal foundation for the human↔AI context-alignment layer of an autonomous research pipeline. The evidence supports a qualified HYBRID, not a clean yes or no. The paradigm's mechanisms are a near-exact structural match for what Solve #2 needs: the dominant tools all model work as a typed, gated state machine (Jira's statuses + one-directional transitions; Linear's fixed status categories) that maps cleanly onto the SSOT's (category, status) contract, and they ship an explicit admission gate (Linear Triage) that prefigures graduated approval. Crucially, the deepest justification for the layer is not the tools at all but the CSCW theory the tools instantiate: cooperative work provably incurs irreducible "articulation work" overhead (Schmidt & Bannon), a shared store is not a shared understanding (the Common Information Space critique), and a versioned SSOT functions as a boundary object — an artifact plastic enough for a human and a differently-conceptualizing agent to share without forcing one interpretation (Star & Griesemer).

But the analogy breaks precisely where research differs from delivery. Tickets assume fixed scope; research intent is non-stationary — the same CSCW theory says formal plans are only "local and temporary closures" that must be continuously re-articulated, which is why the SSOT must be versioned/append-only rather than a static backlog. Task tools optimize throughput of well-specified deliverables; research is exploratory and uncertainty-dominated. And shared-mental-model theory delivers the sharpest warning of all: similarity is not accuracy — a human and an agent can perfectly agree on a wrong model, which is exactly the goalpost-moving failure when the worker can redefine its own success metric. The defensible design therefore adopts the state-machine + admission-gate + objective-cascade + retrospective machinery, but couples it to research-specific invariants the business paradigm never needed: intent owns the yardstick, never the reading; versioned provenance over mutable tickets; and a verification coupling where higher autonomy mandates stronger independent (cross-model) checking. Strongest complements — not replacements — are experiment-tracking / electronic-lab-notebook paradigms (W&B, MLflow) for the reading half and OKR cascades for the objective half.

This report synthesizes 21 independently-verified claims (each survived 3-vote adversarial fact-checking against primary sources) into seven findings. It deliberately separates two layers the research question conflates: the tool layer (Jira / Linear / Asana data models) and the theory layer (CSCW / HCI accounts of why coordination artifacts align people at all). The tool layer answers "what construct do we copy?"; the theory layer answers the harder question — "will copying it actually align a human and an autonomous agent, or just log their work?"

Finding 1

How the paradigm works: every dominant tool is a typed, gated state machineHigh confidence

Stripped of branding, the leading PM tools converge on one data model: a work item is an object that occupies exactly one status at a time, statuses are grouped into ordered categories, and movement between statuses is governed by an explicit set of transitions. This is a finite state machine, and it is the single most transferable construct for the SSOT.

1A The lifecycle is an explicit, gated-by-construction state machine

What it is: Jira defines a workflow as "a set of statuses and transitions that a work item moves through." Statuses are the discrete states; transitions are the directed, one-directional edges between them. An item can move between two statuses only if a transition explicitly connecting them exists — and to move back, a second transition must be declared.
"A set of statuses and transitions that a work item moves through… For a work item to move between two statuses, a transition must exist… Transitions are one-way." — Atlassian, Jira Cloud workflow docs (primary). Verified verbatim; the global-transition feature is itself a declared transition, so the "only declared moves are permitted" invariant holds.
Why it matters for us This is the canonical PM-tool data model that maps directly onto the SSOT's (category, status) state machine and its legality matrix. "Declared-moves-only + one-directional" is the defining property of a finite state machine, which a legality matrix encodes — the base pipeline's reject-before-write transition contract is the same mechanism Jira ships. One nuance: a bare Jira transition gates only source→target; a full legality matrix also keys on op and actor. In Jira that extra gating lives in transition conditions / validators / permissions, not in the transition link itself — so the analogy is to the structural source→target gate, with the op/actor dimension being our addition.

1B The category set is typed and constrained, not free-form

What it is: Linear builds its workflow on a fixed, non-reorderable set of status categories — Backlog, Unstarted, Started, Completed, Canceled, plus a reserved Duplicate. Customization is confined to renaming / adding statuses within a category; users cannot create new categories or reorder the set.
"Statuses can be rearranged within a category but categories cannot be moved around." — Linear, configuring-workflows docs (primary). Verified verbatim.
Why it matters for us This is the typed-state-machine pattern the SSOT's brainstorm / in-progress / success / fail lanes already use: a small, fixed, semantically-meaningful category set with controlled customization underneath. Linear demonstrates that the construct survives real-world product scale precisely because the category vocabulary is locked while sub-statuses flex.

1C Work enters via an explicit admission gate

What it is: Linear's Triage is an optional status that lets a team review issues before accepting them into a backlog or cycle — an explicit accept / decline / mark-duplicate / snooze decision. Triage items sit outside the normal committed workflow until admitted.
"Triage is an optional status in Linear that makes it possible to review issues before accepting them and moving them to your team's backlog or cycles." — Linear, conceptual-model docs (primary). Verified verbatim; Triage is a staging ground for accepting/rejecting, excluded from normal views by default.
Why it matters for us This is a working, shipped instance of graduated admission: an item is not "committed work" until a gate passes. It directly prefigures Solve #2's graduated gating, where the hierarchy level sets human-approval strength — a new direction or a metric change should pass through a Triage-like admission step before it becomes part of the committed objective tree.

Hierarchy: a configurable tree — but not a strict one

Above the single item, the tools model a multi-level hierarchy. Jira's Advanced Roadmaps / Plans defaults to a three-tier Epic > Story > Sub-task chain, with Initiative as an optional fourth tier above Epic for cross-team priorities — the canonical portfolio > epic > story > subtask shape the project > direction > task SSOT analogy rests on. (Two precision notes from verification: Atlassian's own name for the tier above Epic is Initiative, not "Portfolio" — "Portfolio" was the legacy product name; and a Task issue actually sits at the Story level as a sibling, so the precise chain is Epic > {Story / Task / Bug} > Sub-task.)

The tree assumption is leaky — and that is informative

  • Asana breaks the strict tree. A task can be multi-homed across two or more projects: "Tasks can be orphaned and belong to no projects, they can belong to one project, or they can be multi-homed across two or more projects." The data model holds a set of project memberships, not a single parent pointer — a many-to-many graph, not a tree (Asana developer docs, verified).
  • Linear's strict Initiative → Project → Milestone → Issue → Sub-issue nesting was REFUTED (0–3) against the same primary docs — a caution against over-stating any tool's hierarchy as a clean tree.
  • Design implication: a research task may legitimately serve more than one direction (e.g., a shared ablation that informs two hypotheses). The SSOT should decide deliberately whether a task belongs to a single direction or is shareable; the dominant paradigm itself does not mandate strict single-parent nesting.

Takeaways — Finding 1

  1. The transferable core is the typed, gated, one-directional state machine (Jira/Linear), not the branding. It is a structural twin of the SSOT's (category, status) + legality-matrix contract.
  2. Admission gates already exist (Linear Triage) and are the natural home for graduated, hierarchy-level-keyed approval.
  3. The hierarchy is configurable and not necessarily a strict tree — Asana multi-homing shows tasks can be shared, so single-parent nesting is a choice, not a requirement.
Finding 2

Why these artifacts align people — and why a store alone does notHigh confidence

The strongest justification for Solve #2 is not that Jira exists; it is the CSCW theory of why coordination artifacts align cooperating actors at all. Three primary results from the founding CSCW literature (Schmidt & Bannon 1992; Bannon & Bødker 1997) carry directly — with one structural caveat each.

2A Coordination overhead is irreducible — budget for it as a first-class function

What it is: Cooperative work necessarily generates a distinct category of overhead labor — articulation work — to divide, allocate, coordinate, schedule, mesh and interrelate the distributed activities. It is an inherent cost of interdependence, not an accident.
"Because of this interdependence, any cooperative effort thus involves a number of secondary activities… the cooperating workers have to articulate (divide, allocate, coordinate, schedule, mesh, interrelate, etc.) their distributed individual activities… cooperative work implies an overhead cost in terms of labor, resources, time." — Schmidt & Bannon (1992), Taking CSCW Seriously, CSCW 1(1–2):7–40 (primary, foundational).
Why it matters for us A human + autonomous agent is a genuinely interdependent ensemble, so it incurs this overhead. Solve #2 must therefore treat alignment / articulation as an explicit, resourced function — the dashboard, the SSOT, and the autonomy dial are the articulation-support machinery — not as something that happens for free. Caveat (from verification): Schmidt & Bannon studied human–human cooperation; the human–AI extension is a sound design inference, attributable to us, not to them.

2B Plans are only "local and temporary closures" — the theoretical basis for NON-STATIONARITY

What it is: Coordination mechanisms (plans, schedules, procedures) are only local and temporary closures and themselves require ongoing articulation work, because every real-world system is open: no formal plan can anticipate every contingency. Plans must be continuously reconciled, renegotiated, and patched.
"…being nothing but local and temporary closures, these mechanisms themselves require articulation work… It is impossible, both in practice and in theory, to anticipate and provide for every contingency… No formal description of a system (or plan for its work) can thus be complete." — Schmidt & Bannon (1992), citing Gerson & Star (1986). Reinforced by Suchman's Plans and Situated Actions.
Why it matters for us This is the direct theoretical warrant for treating research intent as non-stationary. A fixed-scope backlog ticket is exactly the "complete formal plan" the theory says cannot exist for open work. The SSOT must therefore support continuous re-articulation of objective / metric / gate — versioned and append-only — and the system must support the second-order work of maintaining the mechanisms themselves (revising the metric, not just the result). This is the single sharpest reason the business paradigm cannot be adopted wholesale.

2C A shared store is not a shared understanding — the Common Information Space

What it is: The second core support requirement (alongside workflow management) is constructing a Common Information Space (CIS): a central archive that distributed actors maintain with locally-constructed shared agreement about meaning — even though items differ in origin/context and actors employ divergent conceptualizations. Critically, a CIS is NOT reducible to a shared database.
"Cooperative work is not facilitated simply by the provision of a shared database, but requires the active construction by the participants of a common information space where the meanings of the shared objects are debated and resolved, at least locally and temporarily." — Bannon & Bødker (1997), Constructing Common Information Spaces, ECSCW'97 (primary), restating Schmidt & Bannon (1992).
Why it matters for us This is the explicit caution that a versioned SSOT (a store) does not by itself produce human–agent alignment. Meaning must be actively constructed on top of the stored objects. The dashboard's job is shared-meaning maintenance — surfacing the agent's interpretation of state, claims, and evidence — not merely persisting rows. A human and a differently-conceptualizing agent are the textbook CIS case.

2D Sharing costs bidirectional effort: PACK and UNPACK

What it is: Establishing the shared space costs real, two-sided labor. The producer must PACK information (package an item so a future, possibly anonymous reader can decode it); the consumer must UNPACK it (make sense of it, often lacking the production context). This is articulation work that designers wrongly assume away.
"There is effort expended on the part of people who put information into the CIS… how they package the information… so that it can be successfully decoded by other, perhaps anonymous people… There is also effort expended on the part of those people who lookup information… when they may lack knowledge of the context in which the information was produced." — Bannon & Bødker (1997), ECSCW'97 (primary). The paper frames this added work explicitly as articulation work.
Why it matters for us Solve #2 has two packing/unpacking surfaces, both of which must be budgeted: (a) model→user — the agent must PACK its state/claims/evidence so the human can UNPACK without re-reading raw logs (this is the dashboard's reason to exist); (b) user→model — the human must PACK intent into the Task/Objective surface so the agent can UNPACK it into actionable scope. Neither direction is free; an under-designed dashboard or objective surface silently shifts cost onto the other party.

A neighbouring claim was REFUTED — do not over-license the typed SSOT from this source

  • A claim that Schmidt & Bannon "directly license a task-management-style typed SSOT (hierarchy, status workflows, prioritized backlog, classification schemes) as the coordination substrate" was REFUTED (1–2). The theory supports treating coordination artifacts as material, explicitly-represented mechanisms, but stops short of endorsing the full Jira-style toolkit as the substrate. The justification for the typed SSOT comes from the tool evidence (Finding 1) and the boundary-object construct (Finding 3) — not from a blanket reading of articulation-work theory.

Takeaways — Finding 2

  1. Alignment is never free (articulation work is irreducible); the dashboard + SSOT + dial are the articulation-support function and must be resourced as such.
  2. Plans are temporary closures — this is the theoretical bedrock for a versioned, append-able, re-articulable SSOT and rules out fixed-scope tickets for research intent.
  3. A store ≠ alignment. Meaning is constructed on top of objects; the dashboard must surface interpretation, and both PACK and UNPACK effort must be designed for in both directions.
Finding 3

The boundary object & common ground: how a human and a differently-thinking agent can share one artifactHigh confidence

If a store is not alignment, what artifact does let two parties with different internal models coordinate? Two convergent theory lineages answer this: boundary objects (STS / Star & Griesemer) and common ground / grounding (psycholinguistics / Clark & Brennan). Both reach the same operationally crucial conclusion for an autonomous agent: alignment does not require an identical shared model — it requires a shared artifact and purpose-relative sufficiency.

3A The SSOT as boundary object: plastic yet identity-preserving

What it is: A boundary object has two simultaneous properties: plastic enough to adapt to the differing local needs of multiple parties, yet robust enough to maintain a common identity across all of them.
"…both plastic enough to adapt to local needs and constraints of the several parties employing them, yet robust enough to maintain a common identity across sites." — Star & Griesemer (1989), Institutional Ecology, 'Translations' and Boundary Objects, Social Studies of Science 19(3):387–420, p.393 (primary, foundational).
Why it matters for us The versioned project → direction → task SSOT can serve both an autonomous agent and a human supervisor without forcing one shared interpretation: the human reads it as a north-star + approval surface; the agent reads it as an executable scope/gate spec. Same object, different local meanings, one identity. This is the construct that makes a single SSOT viable across two very different "social worlds."

3B Cooperation WITHOUT consensus — and the two translation mechanisms

What it is: Heterogeneous actors with divergent viewpoints can cooperate without prior consensus on meaning or a single shared representation. Star & Griesemer name two major activities central to translating between viewpoints: standardization of methods, and the development of boundary objects.
"…two major activities are central for translating between viewpoints: standardization of methods, and the development of 'boundary objects'." — Star & Griesemer (1989), abstract (primary). NB: "two major activities," scoped to their case study — not literally "the only two."
Why it matters for us This pairs almost one-to-one with the design: standardized methods → the autonomy dial's required, standardized (cross-model) verification; boundary object → the versioned SSOT. The human and agent need not hold an identical task model, only a shared boundary artifact plus standardized verification methods. (Verification note: Star's "standardization of methods" originally means standardized data-collection protocols/forms, which maps as well or better to the SSOT's typed schema as to the approval dial — both readings are defensible.)

The masking-of-asymmetry critique is directly germane to OUR goalpost-moving risk

  • The boundary-object critique literature (Huvila 2011; Oswick & Robertson 2009; Star's own 2010 "This Is Not a Boundary Object") warns the construct is not politically neutral — interpretive flexibility can mask or entrench power asymmetry rather than dissolve it.
  • Translated to our system: the same plasticity that lets a human and agent share the SSOT is exactly what an agent could exploit to reinterpret success — quietly redefining the metric under cover of "interpretive flexibility." This is why the SSOT invariant intent owns the yardstick, never the reading is load-bearing: the boundary object may flex in interpretation but the metric definition must be pinned by the human-owned intent layer.
  • Designer's obligation (Star 2010): specify the SSOT's information-structure / work-process arrangements, not just assert its interpretive flexibility.

Common ground: alignment is a continuous process, calibrated to "current purposes"

3C Common ground & grounding

What it is: Common ground is the mutual knowledge, beliefs, and assumptions shared by participants; all collective action requires updating it moment-by-moment via a process Clark & Brennan call grounding.
"…without assuming a vast amount of shared information or common ground—that is, mutual knowledge, mutual beliefs, and mutual assumptions… to coordinate on process, they need to update their common ground moment by moment. All collective actions are built on common ground and its accumulation." — Clark & Brennan (1991), Grounding in Communication (primary, ~4,665 citations).
Why it matters for us A dashboard updated once is not alignment; common ground must be continuously re-grounded. This is the psycholinguistic counterpart to Finding 2B's "temporary closures" — both say the same thing from different fields: the alignment surface must be live and real-time, not a one-shot brief.

3D The grounding criterion: "good enough for current purposes," not perfect

What it is: The grounding criterion is that participants seek mutual belief that the addressee has understood the contributor's meaning well enough for current purposes — a purpose-relative sufficiency threshold, not absolute understanding. The threshold itself shifts as collective purposes change.
"…that the partners have understood what the contributor meant to a criterion sufficient for current purposes. This is called the grounding criterion." — Clark & Brennan (1991), p.129 (primary). "Of course, understanding can never be perfect."
Why it matters for us This justifies the autonomy dial as a purpose-relative grounding threshold. Supervised mode demands a high grounding criterion (frequent confirmation that the human understood the agent's plan); Autonomous mode relaxes per-step grounding but raises the verification bar instead. The dial is, in grounding terms, a knob on "how sufficient is sufficient for current purposes" — and because the threshold legitimately shifts with purpose, a per-task dial is theoretically well-founded, not ad hoc.

3E Grounding uses positive evidence — and the medium changes its cost

What it is: Grounding is achieved via three forms of positive evidence of understanding — acknowledgments ("uh huh"), initiation of the relevant next turn, and continued attention. And the medium systematically changes the cost/availability of these techniques along eight dimensions: copresence, visibility, audibility, cotemporality, simultaneity, sequentiality, reviewability, revisability.
"…people ultimately seek positive evidence of understanding…"  |  "…what takes effort changes dramatically with the communication medium… 1. Copresence… 7. Reviewability… 8. Revisability." — Clark & Brennan (1991) (primary, both verified verbatim).
Why it matters for us The choice of asynchronous dashboard vs. live channel is not cosmetic — it changes which grounding mechanisms are even possible. An async text dashboard is high on reviewability and revisability (you can re-read and edit the SSOT) but low on cotemporality / simultaneity (no precisely-timed acknowledgment). Solve #2's bidirectional surface should lean into the dimensions async text is good at — reviewable, revisable, persistent claims+evidence — and explicitly compensate for the timing-dependent evidence it cannot cheaply provide (e.g., the agent posting structured "relevant next turn" updates as its proxy for grounding when the human is away). The autonomy dial's Async/Autonomous settings are, in these terms, a deliberate trade of cotemporality for reviewability.

Takeaways — Finding 3

  1. The versioned SSOT is justifiable as a boundary object: it lets a human and a differently-conceptualizing agent share one artifact without a single forced interpretation.
  2. Alignment needs no identical shared model — only a shared artifact + standardized methods + purpose-relative grounding. The autonomy dial is a grounding-criterion knob.
  3. The medium constrains grounding: async dashboards win on reviewability/revisability and lose on timing — design to the medium's strengths.
  4. Boundary-object plasticity is double-edged: it is also the attack surface for goalpost-moving, which the yardstick-vs-reading invariant must close.
Finding 4

Shared mental models: the mechanism that aligns — and the trap that should terrify usHigh confidence

Why would a shared task/objective surface align a human and an agent rather than merely log work? Shared-mental-model (SMM) theory supplies the mechanism — and, in the same breath, the single most important warning in this entire report.

4A The convergence-to-prediction mechanism

What it is: When teammates hold similar mental models of the shared task and of each other, their models converge so they arrive at equivalent predictions of the task environment and can accurately predict each other's needs and actions.
"…if team members have similar mental models of their shared task and of each other, then they are able to accurately predict the needs and actions of their teammates… mental models tend to converge. As a result, individuals arrive at equivalent predictions of the task environment." — theoretical review of SMMs in human–AI teams (2022), Theoretical Issues in Ergonomics Science, building on Cannon-Bowers, Salas & Converse (1993) (primary review).
Why it matters for us This is the theoretical basis for why an agreed SSOT + dashboard would align a human and an agent rather than merely record activity: a shared, mutually-held task model lets each party predict the other, enabling anticipatory coordination. It is what a backlog cannot do by being a backlog — the value is in the shared model the surface induces, not the rows.

4B Similarity is NOT accuracy — the goalpost-moving theorem

What it is: Sharedness of a mental model is not the same as its correctness. A team can perfectly share a model that is wrong; inaccurate shared models lead to poor prioritization and strategy. Empirically, accuracy is often a stronger predictor of performance than similarity (Edwards et al. 2006; McIntyre & Foti 2013).
"…all team members could have similar knowledge and they could all be wrong—highlighting that similarity alone is insufficient without accuracy." — Cooke et al. (2000), as quoted in the 2022 human–AI SMM review (primary). Corroborated by multiple replications.
Why it matters for us This is the formal statement of the goalpost-moving risk. If the agent can redefine its own success metric, it can drive the human–agent pair to a perfectly shared but wrong model — full agreement on a metric that no longer measures the real objective. Shared agreement alone does not guarantee correctness. This directly substantiates the SSOT invariant: the intent layer must own the yardstick (the agreed model/metric) separately from the reading (results), and the agent must be structurally barred from editing the yardstick. It is also why higher autonomy must mandate stronger independent verification — an external (cross-model) check is the accuracy anchor that similarity cannot supply.

4C Taskwork vs teamwork — and what is still unsolved for human–AI pairs

What it is: Mental models decompose into taskwork (procedures, tasks, equipment) and teamwork (roles, interaction patterns, information flow). How the SMM construct works for human–AI teams — whether/how AI agents form SMMs and how to quantify them — is an explicitly open question lacking consistent terms and valid measurement.
"…it remains unclear how the SMM construct may differ in teams of human and AI agents, how and under what conditions such SMMs form, and how they should be quantified… a lack of consistency in terms and of effective means for measuring Human-AI SMMs currently impedes realization of the concept." — 2022 human–AI SMM review (primary). Status persists in 2024–2025 literature; no source claims it solved.
Why it matters for us This bounds the entire analogy. Business task-management surfaces capture taskwork well (that is exactly what a ticket is). But the teamwork dimension — and any bidirectional measurement of whether a human and an agent actually share a model — remains unsolved. So adopting the PM paradigm wholesale does not automatically deliver a measurable shared model with an autonomous agent. Solve #2 can build the surface, but cannot assume the SMM forms; it must instrument for it (this is an open question, below).

Takeaways — Finding 4

  1. SMM theory explains why a shared surface aligns (convergence-to-prediction) — the payoff is the shared model, not the rows.
  2. Similarity ≠ accuracy is the goalpost-moving theorem: a shared-but-wrong model is the central failure, mandating yardstick≠reading and independent verification scaling with autonomy.
  3. PM surfaces cover taskwork; the teamwork dimension and human–AI SMM measurement are unsolved — the paradigm does not auto-deliver a measurable shared model.

  • Break 1 — non-stationary objectives vs fixed-scope tickets. A ticket assumes scope is set; research revises objective, metric, gate, even architecture. CSCW theory agrees plans are only temporary closures (2B). ⇒ versioned/append-only SSOT, not mutable tickets.
  • Break 2 — exploratory/uncertain work vs deliverable throughput. PM tools optimize flow of well-specified deliverables (WIP limits, cycle time). Research value is information gain under uncertainty; a "fail" lane is a first-class outcome, not a missed deadline.
  • Break 3 — one-or-few researchers vs large teams. Much PM machinery (assignment, capacity planning, multi-team coordination) is overhead for a 1-human-1-agent pair. The relevant coordination is human↔agent articulation (Finding 2), not inter-person scheduling.
  • Break 4 — goalpost-moving when the worker owns the metric. The similarity≠accuracy theorem (4B) makes this a theorem, not a worry: a shared-but-wrong metric is the predicted failure mode. PM tools never had to defend against the worker redefining "done," because the human always owned the ticket.

Takeaways — Finding 5

  1. Structural mappings are strong (hierarchy, state machine, dial, gating); the breaks cluster on the four research-specific properties — non-stationarity, exploration, small teams, goalpost risk.
  2. Every break is repairable by extension, not by abandoning the paradigm: version the transitions, treat fail as a real outcome, drop person-coordination machinery, and pin the yardstick.
  3. The OKR cascade and retrospective mappings are plausible design recommendations, weaker in direct evidence than the state-machine mappings — flag them as such.

OKR objective→key-result cascade ADOPT (modified) Use for project→direction→task objective alignment — BUT the key-result/metric is owned by the human-intent layer (the yardstick). The agent contributes readings, never edits the KR definition. [invariant: yardstick≠reading]
Retrospectives / sprint reviews ADOPT (modified) Map to the self-learning + self-reflection loop: distill durable rules from each package's outcome. Keep rule-distillation manual/gated, not auto-populated from results (matches the base pipeline's Rules/Insights design). [F5]
Hierarchy (portfolio→epic→story→subtask) ADOPT (modified) Adopt as project→direction→task, but allow a task to serve multiple directions (Asana multi-homing) rather than forcing a strict tree. Version every node. [F1, F2B]
Boards / dashboard ADOPT (modified) Reframe from a status board (a store) to a Common Information Space: surface the agent's interpretation, claims, and evidence; design PACK/UNPACK both directions; re-ground continuously. [F2C, F2D, F3C]
Mutable in-place tickets REJECT Intent is non-stationary; in-place mutation destroys the provenance needed to audit a moved goalpost. Replace with versioned/append-only nodes. [F2B, F4B]
Worker-owned "definition of done" REJECT The single most dangerous import. Similarity≠accuracy means a self-defined metric enables a shared-but-wrong model. The yardstick lives in the human-intent layer; the agent supplies readings only. [F4B]
Throughput / velocity / capacity metrics REJECT Optimizing deliverable throughput is the wrong objective for exploratory work; rewards motion over information gain and penalizes the (essential) fail lane. [F5 Break 2]
Assignment / multi-team scheduling / capacity planning REJECT Overhead for a 1-human-1-agent pair; the real coordination is human↔agent articulation, not inter-person allocation. [F5 Break 3]

The integration principle in one line

Import the state machine, the admission gate, the objective cascade, the activity log, and the retrospective — then bolt them to research invariants the business paradigm never needed. The business paradigm gives you a faithful, battle-tested chassis for the taskwork and the gated transitions; it gives you almost nothing for non-stationarity, for the fail-as-outcome economy of exploration, or for defending the yardstick against a self-interested worker.

Yardstick ≠ readingIntent layer owns the metric definition; agent owns only the measured value. Closes the goalpost-moving attack surface (F4B) and the boundary-object masking critique (F3).
Versioned > mutableAppend-only, provenance-bearing transitions because plans are temporary closures (F2B). Makes a moved goalpost auditable.
Gate strength = levelTriage-style admission (F1C) with approval strength rising up the hierarchy; coupled to the autonomy dial as a grounding-criterion knob (F3D).
Autonomy ⇒ verificationIndependent (cross-model) verification scales with autonomy, supplying the accuracy anchor similarity cannot (F4B). This coupling is ours, not the PM paradigm's.
Finding 7

Verdict & the strongest alternativesHigh confidence (direction) / medium (alternative rankings)

Is it optimal to base Solve #2 on the business task-management paradigm? → HYBRID, leaning yes-for-the-chassis / no-for-the-philosophy.

The business paradigm is the right structural foundation for the gated, typed, hierarchical mechanics of the alignment layer — its state machine, admission gate, and hierarchy are battle-tested and map cleanly. But it is not a sufficient foundation, and a naive adoption is actively dangerous, for three reasons each grounded in verified theory:

  • (1) Its core metaphor is delivery throughput; research is exploration. The optimization target is wrong, and the fail-as-outcome economy of research has no PM analogue (Break 2).
  • (2) It assumes fixed scope; research intent is non-stationary. CSCW theory says formal plans are only temporary closures (F2B) — a backlog ticket is the very thing the theory says cannot exist for open work.
  • (3) It never had to stop the worker redefining "done." Similarity≠accuracy (F4B) makes a self-defined metric a predicted failure, not a hypothetical — the yardstick must be removed from the agent's reach.

Hence: adopt the mechanisms, govern them with research invariants, and complement — not replace — them with paradigms that cover what PM tools structurally cannot. The PM paradigm owns the taskwork chassis; the complements own the reading and the objective.

Bottom line

  1. HYBRID. Base Solve #2's mechanics on the PM state-machine/hierarchy/admission-gate chassis — it is verified, transferable, and battle-tested.
  2. Govern that chassis with research invariants the paradigm never needed: yardstick≠reading, versioned/append-only provenance, gate-strength-by-level, autonomy⇒independent-verification.
  3. Complement, do not replace: experiment-tracking/ELN for the reading, OKR for the objective cascade, mixed-initiative HCI for the dial. None alone is sufficient; the PM chassis + these complements + CSCW-grounded re-grounding is the optimal composite.
Appendix A

Caveats, confidence & open questions

Caveats & time-sensitivity

  • Theory layer is rock-solid; tool layer is current but mutable. The CSCW/HCI claims (Schmidt & Bannon, Star & Griesemer, Clark & Brennan, SMM review) are foundational, peer-reviewed, and time-invariant. The tool claims (Jira, Linear, Asana data models) are verified against current primary vendor docs (2025–2026) but are product features that can change — e.g., Asana multi-homing is being actively enhanced, Linear Triage gained Intelligence/Rules. Re-verify tool specifics before implementation.
  • Human–AI extensions are our inferences, not the sources'. Articulation work, boundary objects, common ground, and SMMs were all theorized for human–human cooperation. Transferring them to a human↔autonomous-agent pair is a defensible design analogy but is attributable to this report, not to the cited authors. The agent-as-genuine-teammate premise is the load-bearing assumption.
  • Two refuted claims bound the argument. (a) Schmidt & Bannon do not directly license a full Jira-style typed SSOT as "the" substrate (REFUTED 1–2) — that warrant comes from the tool + boundary-object evidence instead. (b) Linear's strict 5-level nesting was REFUTED (0–3) — do not over-claim any tool's hierarchy as a clean tree.
  • OKR and retrospective mappings are weaker in this corpus. They are widely-recognized methodologies but were not the subject of verified primary-source claims here; they are carried as reasoned design recommendations (medium confidence), not as verified facts.
  • Some paywalled sources confirmed via index, not direct fetch. The SMM review (tandfonline 403) and the Schmidt & Bannon PDF host (ECONNREFUSED) were triangulated across multiple independent indexes; quotes were corroborated verbatim but not always from a single live page fetch.

Open questions

  1. How do you measure whether a human and an autonomous agent actually share a mental model? The 2022 review states bidirectional human–AI SMM measurement is unsolved and lacks valid instruments (F4C), and 2024–2025 work confirms the gap persists. Solve #2 can build the surface but cannot yet verify the shared model forms — what proxy instrumentation (e.g., agent-predicts-human-decision tests) would close this?
  2. What is the exact mechanism that pins the yardstick yet preserves boundary-object plasticity? The SSOT must be plastic enough to serve human and agent differently (F3A) yet rigid enough that the metric definition cannot drift (F4B). The precise schema-level split between the immutable yardstick and the flexible interpretation is an unresolved design question.
  3. How should the grounding criterion (autonomy dial) re-tighten automatically when purpose shifts? Clark & Brennan say the sufficiency threshold legitimately changes as collective purposes change (F3D). When a non-stationary objective is revised, should the dial auto-revert toward Supervised until re-grounded? What triggers that?
  4. Which CIS/grounding mechanisms can an async dashboard actually deliver for a human who is away? The medium-constraints result (F3E) says async text trades cotemporality for reviewability. What is the minimal set of "positive evidence of understanding" the agent must PACK into the dashboard so an absent human can re-ground cheaply on return — and does the present-only Question-Tool ack model satisfy it?
