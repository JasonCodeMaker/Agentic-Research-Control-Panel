The Anatomy of Auto-Research Frameworks: Common Paradigm, Harness & Workflow

A source-level survey of eight open auto-research frameworks — studied directly from their code and skill files — that reverse-engineers the single shared architecture hiding under their wildly different surfaces.

gpt-researcher academic-research-skills AI-Research-SKILLs ARIS (research-in-sleep) AutoResearchClaw Scientific Agent Skills superpowers autoresearch

Abstract / TL;DR

An auto-research framework is an agentic system that takes an under-specified human intent — a topic, a query, a brief, or a coding request — and drives it autonomously toward a research deliverable: a cited report, a publishable paper, a trained model, or shipped code. This survey reads eight open frameworks at the source level, spanning a Python web-research engine, four Claude-Code skill libraries, two coded experimental pipelines, and a minimal single-file training loop. The headline finding is convergence: strip away Python pipelines versus Markdown skills, single-agent loops versus 37-agent panels, and all eight decompose “research” into the same seven functional roles (Scope, Search/Read, Ideate, Experiment, Verify, Synthesize/Write, Remember). They host those roles in one of two harness styles — a hard-coded stage machine or a skill-triggered self-orchestrating agent — and they walk the same canonical workflow spine: Scope → Plan → Gather → Process → Synthesize → Verify → Report, wrapped in an iteration loop. What actually differs is not which parts exist but how heavily each is instantiated and where the human and the verifier sit. Across the corpus the same load-bearing principle recurs almost verbatim — “a loop can DRIVE but it cannot ACQUIT” — the producer of work must never be the sole judge of it. The reusable unit is the function set, not the agent count.

Section 02

Common Paradigm: Functional Building Blocks

Strip away the surface differences — Python pipelines vs. Markdown skills, single-agent loops vs. 37-agent panels, web research vs. GPU training — and all eight frameworks decompose “research” into the same small set of functional roles. They are not arbitrary; they trace the shape of the scientific method itself: narrow the question, find what's known, propose something new, test it, check whether the test was honest, write it up, and remember what you learned. A builder who understands these roles as reusable primitives can read any of these systems as a particular wiring of the same parts, and compose a new system by choosing which roles to include and how heavily to instantiate each.

The interesting variation is not whether a role exists but how it is implemented and how much of the lifecycle a framework chooses to cover. The same “Verifier” appears as a single immutable evaluate_bpb() function in Karpathy's autoresearch, a whitelist-of-allowed-numbers in AutoResearchClaw, and a five-layer cross-model adversarial audit chain in ARIS. The same “Ideator” is one inline code edit in autoresearch and a 10-framework diverge-converge dialogue in AI-Research-SKILLs. Below are the seven recurring roles, each named as a primitive, with the frameworks that exhibit it and how their implementations diverge.

R1 Scoper / Decomposer — turning a vague prompt into a structured problem

What it is. Almost every framework inserts a front-stage that converts a one-line topic into a typed, bounded problem — research questions, sub-problems, constraints, explicit non-goals — before any search or generation. This fixes scope drift at the source and gives downstream roles a stable contract. Outputs range from a free-text constraint file to a FINER-scored research question to a recursive sub-query tree.
  • gpt-researcher — plan_research_outline() decomposes the query into N sub-queries each with a goal; choose_agent() additionally classifies the domain into a role-prompt. gpt_researcher/actions/query_processing.py
  • academic-research-skills — research_question_agent produces FINER-scored RQs with scope boundaries; intake_agent emits a Paper Configuration Record consumed downstream. academic-paper/agents/intake_agent.md
  • AutoResearchClaw — Stages 1–2 (TOPIC_INIT, PROBLEM_DECOMPOSE) build a structured problem tree of research questions and sub-problems. researchclaw/pipeline/stage_impls/_topic.py
  • ARIS — Phase 0 reads RESEARCH_BRIEF.md to extract problem, constraints, domain knowledge, and non-goals before search. skills/idea-discovery/SKILL.md
  • autoresearch — minimalist extreme: scope is a human-edited Markdown constraint file (only edit train.py, no new packages) — the Scoper is a static document, not an LLM call. autoresearch/program.md
  • superpowers — brainstorming runs Socratic one-question-at-a-time scoping and writes an approved spec doc before any implementation skill can fire (HARD-GATE). skills/brainstorming/SKILL.md
Why it matters — Scoping is the first and cheapest defense against context pollution and goal drift: a typed problem statement is what every later role validates against. Frameworks that skip it push ambiguity downstream where it is far more expensive to detect.

R2 Searcher + Reader — the retrieve-then-extract pair

What it is. Information-centric frameworks split knowledge acquisition into two composable primitives: a Searcher that federates many sources to return candidate links, and a Reader/Scraper that fetches full content and distills it. They are almost always a pluggable backend layer with deduplication, so adding a source is configuration, not code. Training-loop frameworks collapse or omit this pair because their evidence comes from experiments, not literature.
  • gpt-researcher — 15+ pluggable retriever backends run via asyncio.to_thread, dedup by a visited-URL set; BrowserManager dispatches to one of several scraper backends. gpt_researcher/skills/researcher.py
  • scientific-agent-skills — paper-lookup federates 10 academic DBs and database-lookup covers 78; parallel-cli extract is the Reader. skills/paper-lookup/SKILL.md
  • academic-research-skills — bibliography_agent runs corpus-first search with Semantic Scholar fallback; a separate knowledge-extract step reads the screened papers. docs/ARCHITECTURE.md
  • AutoResearchClaw — Stages 3–6 separate SEARCH (OpenAlex/S2/arXiv with circuit-breaker degradation) from KNOWLEDGE_EXTRACT (LLM knowledge cards into a per-stage KB). researchclaw/literature/search.py
  • ARIS — /research-lit fans out across arXiv/S2/OpenAlex/Exa under Policy D2, then gates every paper through deterministic verify_papers.py before wiki ingest. docs/SKILLS_CATALOG.md
  • AI-Research-SKILLs — Bootstrap directs Exa MCP + Semantic Scholar + arXiv + CrossRef and persists each paper to literature/survey.md for reuse. 0-autoresearch-skill/SKILL.md
Why it matters — Making retrieval a pluggable backend layer is the single biggest reuse lever a builder gets: the same orchestration survives swapping Tavily for PubMed for an MCP server. The retrieve/read split also localizes where grounding happens.

R3 Ideator / Hypothesizer — generating candidates the system will test

What it is. The creative role that produces what gets tested next: hypotheses, research ideas, or the next code change. Its weight varies more wildly than any other role — from a structured multi-round divergence engine with a banlist of prior failures down to collapsing entirely into the Experimenter. A recurring sophistication is feeding the system's own failed ideas back in as anti-repetition input.
  • AI-Research-SKILLs — two dedicated skills: 10 diverge-converge frameworks (tension hunting, abstraction ladder) plus cognitive-science framings (bisociation, analogical reasoning). 21-research-ideation/brainstorming-research-ideas/SKILL.md
  • ARIS — /idea-creator brainstorms 8–12 ideas via GPT-5.5, reads the wiki query_pack to treat failed ideas as a banlist, then writes both kept and killed ideas back. skills/idea-discovery/SKILL.md
  • scientific-agent-skills — splits the role: scientific-brainstorming (open-ended) vs hypothesis-generation (8-step, needs observations) vs hypogenic (data-driven). skills/hypothesis-generation/SKILL.md
  • AutoResearchClaw — HYPOTHESIS_GEN generates falsifiable hypotheses via multi-perspective LLM debate (_multi_perspective_generate). researchclaw/pipeline/stage_impls/_synthesis.py
  • gpt-researcher — implicit ideation: deep_research() extracts follow-up questions per level and recurses — generation of new sub-questions, not hypotheses. gpt_researcher/skills/deep_research.py
  • autoresearch — collapsed extreme: “Idea Generation + Code Edit” is one step; the agent proposes a train.py change and edits it directly. autoresearch/program.md
Why it matters — Whether ideation is a first-class staged artifact or fused into coding determines how steerable and auditable the research direction is. Treating it as a primitive — especially one that consumes the failed-idea memory — is what turns a random walk into directed search.

R4 Experimenter / Coder — producing and running the executable test

What it is. The role that writes code, runs it, and captures results — present in every framework doing empirical (not just literature) research. Implementations span a single hand-edited file, a sandboxed AST-validated code agent, and an SSH-to-GPU bridge with OOM retry. A near-universal sub-rule is a cheap sanity check (convergence, NaN/Inf, baseline reproduction) before the expensive run, plus a self-heal loop on failure.
  • autoresearch — the agent IS the experimenter: edits train.py, commits, runs a fixed 300s budget; generation and coding collapse into one role. autoresearch/program.md
  • AutoResearchClaw — Stages 9–13 use a multi-phase AST-validated CodeAgent, sandbox/Docker/SSH/domain executors, and self-heal for up to 10 iterations. researchclaw/pipeline/stage_impls/_execution.py
  • ARIS — /experiment-bridge gets a cross-model code review BEFORE GPU time, runs smallest-config sanity, then deploys via SSH+screen, auto-routing by job count. skills/research-pipeline/SKILL.md
  • AI-Research-SKILLs — the inner loop git-commits the protocol BEFORE running, invokes one of 90 domain skills, and mandates sanity checks before trusting results. 0-autoresearch-skill/SKILL.md
  • superpowers — a fresh implementer subagent per task implements with mandatory TDD (failing test first) and reports a typed status — the coder primitive for software. skills/subagent-driven-development/implementer-prompt.md
  • scientific-agent-skills — 70+ package skills (scanpy, rdkit, pydeseq2) each supply curated API docs so the agent writes correct domain code first-try. skills/scanpy/SKILL.md
Why it matters — This is where most hallucination risk turns into measurable consequence, so the surrounding guardrails (pre-registration commits, sanity checks, sandboxes, code review before compute) are as load-bearing as the coding. The cheap-check-before-expensive-run pattern is a reusable cost-and-trust primitive independent of domain.

R5 Verifier / Critic / Jury — the gate that decides whether output can be trusted

What it is. The most architecturally diverse and most emphasized role in the corpus: an independent check that gates progress or publication. It appears in five styles — (a) a single immutable ground-truth metric, (b) a deterministic whitelist/structural rule, (c) an LLM-as-judge against rubrics, (d) a cross-model adversarial jury, (e) a multi-stage review chain. The strongest designs share one principle: the thing being checked must not be the thing doing the checking.
  • autoresearch — verification is one immutable evaluate_bpb() marked “DO NOT CHANGE”; keep/discard is a pure metric comparison the agent cannot reinterpret. autoresearch/prepare.py
  • AutoResearchClaw — VerifiedRegistry builds a whitelist of ground-truth numbers; paper_verifier.py rejects any unverified number in Results sections (deterministic reject-before-write). researchclaw/pipeline/verified_registry.py
  • ARIS — cross-model jury: a loop can DRIVE but cannot ACQUIT — every gate routes to a fresh GPT-5.5 thread that reads artifacts cold; 5-layer audit chain at submission. skills/shared-references/acceptance-gate.md
  • academic-research-skills — integrity_verification_agent runs a 7-mode AI-failure checklist that BLOCKS on any SUSPECTED mode, plus a 7-agent review panel with a scored Devil's Advocate. academic-pipeline/references/ai_research_failure_modes.md
  • AI-Research-SKILLs — ARA Rigor Reviewer scores 6 epistemic dimensions 1–5 (falsifiability, scope, exploration integrity) — but the same model reads its own artifact (weaker independence). 22-agent-native-research-artifact/rigor-reviewer/SKILL.md
  • gpt-researcher — ReviewerAgent checks each draft against guidelines and loops to ReviserAgent until None — a same-model critique loop, no cross-model independence. multi_agents/agents/reviewer.py
  • superpowers — iron law: no completion claim without running the verification command and reading its output in the same message; the spec reviewer is told to distrust the implementer's report. skills/verification-before-completion/SKILL.md
Why it matters — This role is the corpus's center of gravity for trustworthiness. Its divergence is the clearest design fork: cheap-and-rigid (immutable metric / whitelist) scales but only works where ground truth is computable; expensive-and-general (cross-model jury / panel) covers subjective quality but costs more and risks collusion when the checker shares the executor's model family.

R6 Synthesizer / Writer + Reviser — assembling and revising the deliverable

What it is. The pair that turns accumulated evidence into a human-readable artifact and then iterates it. Synthesis clusters and integrates findings (often a separate primitive from writing); the Writer produces the document; a Reviser closes the review→revise loop. A shared discipline across literature-heavy frameworks is that the Writer may only use grounded inputs — session material or a verified-numbers registry — never parametric memory.
  • gpt-researcher — write_report() synthesizes accumulated context into Markdown; the multi-agent layer adds a ReviserAgent that re-enters the reviewer node in a loop. gpt_researcher/skills/writer.py
  • academic-research-skills — draft_writer_agent writes under an Anti-Leakage Protocol — session-material-only fill, flagging gaps as [MATERIAL GAP] rather than hallucinating. academic-paper/references/anti_leakage_protocol.md
  • AutoResearchClaw — PAPER_DRAFT writes a 5–6.5k-word draft driven only by VerifiedRegistry-gated real metrics; PAPER_REVISION adds length and anti-disclaimer guards. researchclaw/pipeline/stage_impls/_paper_writing.py
  • ARIS — /paper-writing chains plan→figure→write→compile→improvement-loop; citations fetched from DBLP/CrossRef, never LLM-fabricated. docs/ARIS_INTRO.md
  • AI-Research-SKILLs — cardinal rule: NEVER generate BibTeX from memory — always fetch programmatically; outer-loop synthesis (findings.md) is a separate role. 20-ml-paper-writing/ml-paper-writing/SKILL.md
  • scientific-agent-skills — scientific-writing uses a two-stage outline-then-prose IMRAD process; literature-review handles the synthesis primitive (thematic, not study-by-study). skills/scientific-writing/SKILL.md
Why it matters — Separating synthesis (find the story) from writing (render it) from revising (improve it) lets each be checked independently, and the grounded-inputs-only rule wires the Writer directly into the anti-hallucination stack. The review↔revise loop is the most portable convergence primitive in the corpus.

R7 Memory / Wiki + Self-Learner — durable knowledge across turns and runs

What it is. A persistence layer surviving beyond a single context window — from none, to a flat results log, to a typed knowledge graph, to a self-modifying skill store. Two sub-flavors recur: research-content memory (papers, ideas, experiments) and harness-level self-learning (lessons that rewrite the system's own instructions). A repeated insight: failed ideas/lessons are the most valuable memory — never pruned and fed back as a banlist.
  • ARIS — Research Wiki: 4 entity types + typed edge graph + budgeted query_pack; failed ideas are privileged memory; /meta-optimize self-evolves SKILL.md files but never auto-applies. skills/research-wiki/SKILL.md
  • AutoResearchClaw — EvolutionStore (JSONL, time-decayed lessons) injects past failures as prompt overlays; MetaClaw converts high-severity lessons into persistent SKILL.md files across runs. researchclaw/evolution.py
  • AI-Research-SKILLs — findings.md is file-based memory (Current Understanding, Lessons, Open Questions); ARA Research Manager promotes observations to claims after 3+ sessions. 0-autoresearch-skill/templates/findings.md
  • academic-research-skills — Material Passport (passport.yaml) is a cross-session resume carrier with a SHA-256 reset-boundary ledger — state persistence, but explicitly NOT self-learning. academic-pipeline/SKILL.md
  • scientific-agent-skills — autoskill is the self-learner: mines repeated workflows from screenpipe, clusters them, proposes new SKILL.md files for user-gated promotion. skills/autoskill/SKILL.md
  • autoresearch — floor case: memory is just results.tsv plus git branch history of kept commits — no rules DB, no reflection, no cross-run synthesis. autoresearch/program.md
  • gpt-researcher — no persistent role: state lives only in the in-memory object / LangGraph TypedDict; each run starts fresh. multi_agents/memory/research.py
Why it matters — Memory is the dividing line between a one-shot pipeline and a system that compounds: the frameworks that improve over time all separate durable knowledge from active context, and the best gate self-modification behind a human act. Failed-ideas-as-banlist is the cheapest high-value memory primitive a builder can add.

Takeaways

  1. Convergence is strongest at the endpoints and weakest in the middle: nearly every framework has a Scoper and a Verifier, but the Ideator and Memory roles are the most optional and variably implemented — treat Scoper+Verifier as mandatory primitives and Ideator/Memory as the levers that distinguish a one-shot pipeline from a compounding research system.
  2. The Verifier is the corpus's most-engineered role and its sharpest fork: cheap-rigid checks (immutable metric, numeric whitelist) scale and cannot be gamed but only work where ground truth is computable; expensive-general checks (cross-model jury, 7-mode block + panel) cover subjective quality but collapse toward self-acquittal when the checker shares the executor's model family. The recurring best-practice is the non-self-acquittal rule — “a loop can DRIVE but cannot ACQUIT.”
  3. Role weight, not role presence, is the real differentiator: the same canonical function spans an immutable evaluate_bpb() and a five-layer audit; a one-line code edit and a 10-framework dialogue. The lightest credible system and the heaviest wire the same primitives — proof that the function set, not the agent count, is the reusable unit.
  4. Two anti-hallucination idioms recur across unrelated frameworks and are worth lifting wholesale: grounded-inputs-only writing (session-material-only / verified-numbers registry / never-fabricate-BibTeX) wired into the Writer, and failed-ideas-as-privileged-memory fed back as an ideation banlist — both cheap to add and directly targeting the deception and repetition failure modes.
Section 03

The Harness: Runtime Scaffolding

A harness is the runtime skeleton hosting the research functions and deciding who runs when, what each component sees, and where state lives. The functions are similar across all eight systems; the scaffolding differs. Two axes dominate: control flow (a hard-coded Python pipeline or typed graph vs. one agent self-routing through Markdown skills) and capability packaging (library-as-code vs. the SKILL.md Agent Skills standard, which is why prose systems bolt on deterministic hooks). The shared invariant: the component that generates an artifact must never be the one that accepts it.

H1 Fixed Stage-Machine vs. Skill-Triggered Self-Orchestration

What it is. An explicit coded stage-machine or typed graph (resumable, rigid) vs. a single agent self-routing through Markdown prose (portable, but the controller is the model's own attention — the surface context-pollution attacks).
  • AutoResearchClaw — execute_pipeline() over a 23-element STAGE_SEQUENCE IntEnum with PIVOT/REFINE re-entry. researchclaw/pipeline/runner.py
  • gpt-researcher — a LangGraph StateGraph with typed edges plus an async single-agent path. multi_agents/agents/orchestrator.py
  • Scientific Agent Skills — no central loop: the host agent reads 142 SKILL.md files and chains them itself. scientific-agent-skills/README.md
  • autoresearch — the entire LOOP FOREVER lives in one human-edited Markdown file. autoresearch/program.md

H2 The Agent Skills Standard plus Deterministic Hooks and Validation Gates

What it is. Capabilities packaged as SKILL.md files discovered by description and loaded only when relevant (progressive disclosure, countering context pollution); because invocation is advisory, trust-focused frameworks add non-LLM enforcement via lifecycle hooks and code validators that reject before write — turning “the model should do X” into “the model cannot do not-X.”
  • AI-Research-SKILLs — 98 SKILL.md files with an explicit progressive-disclosure contract. AI-Research-SKILLs/CLAUDE.md
  • Scientific Agent Skills — makes the Agent Skills standard the orchestration primitive itself. scientific-agent-skills/CONTRIBUTING.md
  • superpowers — a SessionStart hook injects the using-superpowers bootstrap so all skills auto-trigger. superpowers/hooks/hooks.json
  • academic-research-skills — a PreToolUse hook fences agents to their directory and denies Bash: zero fail-open by construction. scripts/ars_write_scope_guard.py
  • AutoResearchClaw — VerifiedRegistry whitelists experiment numbers and rejects any LaTeX number not in it. researchclaw/pipeline/verified_registry.py

H3 Context Isolation plus Cross-Family Adversarial Review

What it is. Independent components are denied each other's context: subagents get only the task text, reviewers read the artifact cold from file paths, coded pipelines isolate via per-stage directories. Trust-focused systems extend this to model heterogeneity, mandating a different model family for the verdict so a model never ratifies its own correlated blind spots; ARIS documents a reused reviewer thread inflating scores from 3/10 to 8/10.
  • ARIS — every reviewer call uses a fresh codex thread (file paths only); cross-family mandatory — a loop can DRIVE but cannot ACQUIT. skills/shared-references/reviewer-independence.md
  • superpowers — subagents never inherit history; the spec reviewer must distrust the implementer report and read code independently. skills/subagent-driven-development/spec-reviewer-prompt.md
  • academic-research-skills — three-tier data_access_level isolation plus an opt-in GPT/Gemini verifier; divergence flagged, never averaged. shared/cross_model_verification.md
  • Scientific Agent Skills — narrow cross-model: one model generates figures, Gemini reviews to threshold; else single-model. skills/scientific-schematics/SKILL.md
  • AI-Research-SKILLs — single-model: the ARA Rigor Reviewer scores its own artifact. 22-agent-native-research-artifact/rigor-reviewer/SKILL.md

H4 Filesystem-Native State, Resumability, and Fan-Out with a Singular Bench

What it is. Durable state lives in the filesystem (JSONL, YAML, per-stage dirs, git) with atomic checkpoints for crash-resilient resume, doubling as an audit trail (vector stores absent); parallelism multiplies breadth and degrades to sequential but decouples fan-out from judgment — subagents generate candidates and the merged set funnels into one acceptance step.
  • ARIS — a 3-tier degradation ladder where shards GENERATE but never SCORE; REVIEW_STATE/run_state give DONE/ACCEPTED resume. skills/shared-references/fan-out-pattern.md
  • AutoResearchClaw — checkpoint plus heartbeat; a Bash sentinel auto-restarts the crashed pipeline from checkpoint. AutoResearchClaw/sentinel.sh
  • academic-research-skills — Material Passport YAML with an append-only SHA-256 reset_boundary ledger for cross-session resume. academic-pipeline/SKILL.md
  • AI-Research-SKILLs — purely file-based: research-state.yaml, findings.md, git pre-registration, no vector store. 0-autoresearch-skill/templates/research-state.yaml
  • autoresearch — results.tsv (untracked, survives reset) plus the autoresearch git branch. autoresearch/program.md

Section 04

The Workflow: End-to-End Pipeline

Strip away the surface differences — 23 numbered Python stages here, 98 loadable Markdown skills there, a single Karpathy-style program.md loop elsewhere — and all eight frameworks instantiate the same underlying spine. Each takes an under-specified human intent and pushes it down a recognizable canonical pipeline, wrapped in some form of Iterate/Self-improve loop.

Where they genuinely diverge is not in which stages exist but in three orthogonal choices: (1) how heavily the pipeline is reified — hard-coded stage machine vs. agent-improvised skill-chaining; (2) where the iteration loop lives and what terminates it; and (3) where the human may enter. The clearest fault line runs between gather-then-write document pipelines and experimental research loops that add a Gather→Run experiments→Synthesize sub-cycle and treat a GPU metric, not a reviewer, as ground truth.

The Canonical Auto-Research Pipeline

W1 The Canonical Spine: Scope → Plan → Gather → Process → Synthesize → Verify → Report

What it is. Every framework walks the same ordered backbone: turn intent into a scoped question, decompose, gather raw material, process it into structured state, synthesize a draft, verify it, and emit a rendered deliverable. The stages are universal; only their reification differs — a literal numbered step, an agent-chosen skill, or a single prose instruction.
  • AutoResearchClaw — most literal: a 23-element IntEnum (TOPIC_INIT … CITATION_VERIFY) grouped into 8 named phases A–H mapping one-to-one onto the spine. researchclaw/pipeline/stages.py
  • academic-research-skills — a 10-stage sequential stage-graph (RESEARCH→WRITE→INTEGRITY→REVIEW→REVISE→…) — the spine with verification stages interleaved. docs/ARCHITECTURE.md
  • gpt-researcher — same spine as a sequential async method chain: choose_agent→plan_research→parallel gather→curate→write_report. gpt_researcher/skills/researcher.py
  • scientific-agent-skills — no coded pipeline; the spine is implicit, reconstructed by the agent chaining skills per the workflow sections in each SKILL.md. skills/literature-review/SKILL.md
  • autoresearch — collapsed spine: constraints→edit train.py→run→verify metric→log. The deliverable is a kept git commit, not a report. autoresearch/program.md
Why it matters — Naming the spine lets a designer treat the pipeline as a fixed skeleton and ask the real questions independently: how reified should each stage be, and which can be skipped. It also exposes that the “autonomy” debate is mostly about the iteration wrapper, not the linear backbone.

W2 Reification Spectrum: Hard-Coded Stage Machine vs. Agent-Improvised Skill-Chaining

What it is. Frameworks differ sharply in how much of the pipeline is fixed in code versus assembled at runtime. At one pole, a deterministic loop iterates a fixed stage list. At the other, there is no orchestrator — the agent reads skill descriptions and self-chains them. This is the single biggest architectural divergence, trading determinism/auditability against flexibility/composability.
  • AutoResearchClaw — hard pole: execute_pipeline() iterates STAGE_SEQUENCE calling execute_stage() — no agent discretion over ordering. researchclaw/pipeline/runner.py
  • scientific-agent-skills — soft pole: 142 flat SKILL.md files, “no central orchestrator, no shared state machine” — the hosting agent IS the orchestration layer. scientific-agent-skills/README.md
  • AI-Research-SKILLs — middle: one meta-skill owns the two-loop control flow but routes to 98 domain skills via a static lookup table. 0-autoresearch-skill/SKILL.md
  • ARIS — middle: skill-chaining where each Markdown skill invokes the next via the Skill tool; sequence prescribed in prose, not a graph engine. skills/research-pipeline/SKILL.md
  • autoresearch — soft pole, minimal: “LOOP FOREVER steps 1–9” in Markdown prose — no DAG, the LLM self-orchestrates. autoresearch/program.md
Why it matters — Reification is the lever for the trustworthiness-vs-flexibility tradeoff. A hard stage machine gives deterministic resume points and audit trails but resists novel task shapes; a flat skill library composes trivially but offers no structural guarantee any stage actually ran. Most production frameworks sit in the middle — orchestrated control flow, improvised leaf execution.

W3 The Experiment Sub-Cycle: Gather Becomes Hypothesize → Code → Run → Diagnose

What it is. Frameworks doing experimental research splice an inner loop into the spine between Plan and Synthesize: form a falsifiable hypothesis, write code, run it under a resource constraint, sanity-check, then self-heal or re-run on failure. A recurring discipline is committing the protocol to git BEFORE running, as lightweight pre-registration. Literature-only pipelines skip this entirely.
  • AI-Research-SKILLs — pick hypothesis → write protocol → git-commit it BEFORE running → execute via domain skill → mandatory sanity checks (convergence, baseline reproduction). 0-autoresearch-skill/SKILL.md
  • AutoResearchClaw — Stages 9–13: design → AST-validated codegen → sandbox/Docker/SSH run → ITERATIVE_REFINE self-heals for up to 10 rounds with NaN/Inf detection. researchclaw/pipeline/stage_impls/_execution.py
  • ARIS — /experiment-bridge gets a cross-model code review BEFORE any GPU time (catches ~80% of bugs), runs smallest config, then deploys via SSH+screen with OOM retry. skills/research-pipeline/SKILL.md
  • autoresearch — maximally collapsed: edit train.py → git commit → run fixed 300s budget → grep val_bpb. ~12 experiments/hour. autoresearch/program.md
Why it matters — This is the clean dividing line between “deep-research report generators” and “autonomous scientists.” The sub-cycle introduces ground-truth signal (a metric, a crash) that no LLM reasoning can fake, which fundamentally changes the verification story. Git pre-registration recurring across three independent frameworks signals a community norm forming.

W4 The Iteration Mechanism: Critique-Loop vs. Metric-Gate vs. Wall-Clock Tick

What it is. Every framework loops, but the loop's driver and terminator fall into three families. (a) Critique-loops re-draft until a reviewer accepts. (b) Metric-gates keep or discard against an objective number. (c) Wall-clock ticks fire on a timer independent of phase, so the loop never idles overnight. Some layer all three. The choice determines whether the system optimizes for quality, measurable progress, or sustained runtime.
critique → quality metric → progress tick → runtime
  • gpt-researcher — critique-loop: ReviewerAgent returns feedback or None; non-null routes a conditional edge to ReviserAgent until None (accept). multi_agents/agents/reviewer.py
  • autoresearch — pure metric-gate: lower val_bpb = keep; equal/worse = git reset; plus a simplicity tiebreaker. Runs until the human kills it. autoresearch/program.md
  • ARIS — critique-loop with hard cap: MAX_ROUNDS=4; STOP requires score ≥ 6 AND verdict in {ready, almost}, cross-model assigned; auto-resume via REVIEW_STATE.json. skills/auto-review-loop/SKILL.md
  • AI-Research-SKILLs — wall-clock tick: a /loop 20m or cron fires every 20 minutes independent of phase — “never idles,” declared MANDATORY. 0-autoresearch-skill/references/agent-continuity.md
  • AutoResearchClaw — budgeted recursive-rollback: Stage 15 returns PROCEED/REFINE(→13)/PIVOT(→8), capped at MAX_DECISION_PIVOTS=2 before forced PROCEED. researchclaw/pipeline/runner.py
Why it matters — The iteration mechanism is where “autonomous” is operationalized, and conflating the three is a common error. Critique-loops can inflate scores via narrative accumulation (ARIS documents 3/10→8/10); metric-gates are gameable only if the metric is editable; ticks give runtime but no quality signal. The strongest systems pair a tick with a metric-gate or cross-model critique.

W5 The Gate Pattern: A Mid-Pipeline Verification Stage That Can Block or Roll Back

What it is. Beyond the iteration loop, most frameworks insert explicit gates — checkpoints where machine verification or a fixed rule decides whether the pipeline may advance, must roll back, or is hard-blocked. Gates are distinct from loops: fixed positions, non-negotiable invariants. The strongest make the gate reject-before-write so no bad artifact is ever produced.
  • academic-research-skills — two blocking integrity gates (Stage 2.5, 4.5): the 7-mode checklist blocks on any SUSPECTED mode; Stage 4.5 is zero-tolerance and re-runs from scratch. academic-pipeline/references/ai_research_failure_modes.md
  • AutoResearchClaw — three fixed GATE stages (5, 9, 20) with DECISION_ROLLBACK targets — e.g. LITERATURE_SCREEN reject rolls back to Stage 4. researchclaw/pipeline/stages.py
  • ARIS — submission gate: a 5-layer cross-model audit emits 6-state verdicts; any FAIL/BLOCKED/ERROR blocks Final Report generation. skills/shared-references/assurance-contract.md
  • superpowers — design HARD-GATE: no implementation skill until the user approves the written spec — a front-of-pipeline gate blocking all downstream action. skills/brainstorming/SKILL.md
  • gpt-researcher — softer gate: optional SourceCurator LLM-ranks sources by credibility before synthesis, capping at top-10 — a filter rather than a hard block. gpt_researcher/skills/curator.py
Why it matters — Gates encode “this invariant must hold before we spend more compute or ship.” Placing a verification gate BEFORE the expensive stage (ARIS reviews code before GPU) converts a costly post-hoc failure into a cheap pre-emptive one. The rollback target turns binary pass/fail into a structured retry — what makes long pipelines resumable rather than fragile.

W6 The Human Entry Point: Front/Back-Loaded vs. Per-Gate vs. Async-Report-Only

What it is. Frameworks place the human at radically different points, deliberately. Three modes recur: (a) front/back-loaded — set up, review the morning after, zero interaction during; (b) per-gate approval — mandatory confirmation at every transition; (c) async-report — the agent pushes progress (HTML/PDF/Telegram) and the human redirects optionally, but the loop never blocks. The mode tracks the framework's autonomy philosophy.
  • autoresearch — front/back-loaded: human edits program.md before and runs analysis.ipynb after; during, zero — the agent is forbidden to pause (“NEVER STOP”). autoresearch/program.md
  • academic-research-skills — per-gate: an Adaptive Checkpoint System mandates confirmation after every stage; “the human decides at every gate.” academic-pipeline/SKILL.md
  • AI-Research-SKILLs — async-report: “the human is asleep or busy”; progress HTML/PDF generated to to_human/, loop never blocks. 0-autoresearch-skill/SKILL.md
  • AutoResearchClaw — configurable across the spectrum: six modes plus SmartPause that triggers human entry from a weighted confidence signal — gate placement becomes a tunable. researchclaw/hitl/smart_pause.py
  • ARIS — hybrid: Gate 1/2 checkpoints with AUTO_PROCEED auto-continuing after 10s, but HUMAN_CHECKPOINT upgrades to per-round pause + Feishu push. skills/idea-discovery/SKILL.md
Why it matters — Where the human enters is the central UX and trust decision. Per-gate maximizes control for high-stakes outputs like papers but throttles overnight throughput; async-report maximizes autonomous runtime but demands strong verification and memory. The frameworks converging on a configurable entry point suggest the future is a tunable dial, not a fixed stance.

W7 The Self-Improvement Outer Loop: Render Is Not the End

What it is. Several frameworks add a stage AFTER the deliverable: scan what happened, extract lessons or provenance, and feed them back to improve the next run or the harness itself. This ranges from ephemeral in-session memory to durable cross-session skill synthesis. Crucially, mature designs gate self-improvement — proposing patches but requiring a separate human or cross-model act before they land — to avoid self-poisoning.
  • AutoResearchClaw — EvolutionStore extracts lessons to JSONL; MetaClaw converts high-severity lessons to persistent SKILL.md files injected into the next run (+18.3% robustness in A/B). researchclaw/evolution.py
  • ARIS — /meta-optimize reads hook event logs, proposes SKILL.md diffs gated by cross-model review, stages them under .aris/meta/pending/, and NEVER applies; landing requires a separate human /meta-apply. skills/meta-optimize/SKILL.md
  • AI-Research-SKILLs — post-task ARA Research Manager classifies session events with provenance tags; a maturity tracker promotes observations to claims at 3+ repeats. 22-agent-native-research-artifact/research-manager/SKILL.md
  • scientific-agent-skills — autoskill clusters repeated user workflows and proposes new SKILL.md files — staged in ~/.autoskill/proposed/, requiring human promote.py. skills/autoskill/SKILL.md
  • superpowers — no runtime memory, but applies TDD to the skills themselves: rationalization tables are crystallized memory of past agent failure modes, hand-curated. skills/writing-skills/SKILL.md
Why it matters — The outer loop separates a one-shot pipeline from a system that gets better at this project over time — the personalization promise. The dominant safety pattern: every mature self-improver proposes-but-gates landing, because an ungated loop can poison itself (ARIS cites the “Hermes failure mode” and ships a capture_filter.py against it). Separate the producer of improvements from the authority that applies them.

Takeaways

  1. Convergence on the spine, divergence on the wrapper: all eight agree on Scope→Plan→Gather→Process→Synthesize→Verify→Render; essentially all differentiation lives in how reified the stages are and what drives/terminates the loop. Design the backbone once; spend your design budget on the loop and the gates.
  2. The cleanest taxonomy split is experiment-vs-no-experiment. Frameworks with the Hypothesize→Code→Run→Diagnose sub-cycle get an unfakeable ground-truth signal and lean on metric-gates; literature/document pipelines have no such oracle and must lean on cross-model critique and citation verification. Choose your verification strategy from this split.
  3. The three iteration mechanisms are complementary, not competing — best-in-class systems layer them: a tick supplies runtime, a metric-gate or cross-model critique supplies a trustworthy stop, a fixed gate supplies a non-negotiable invariant. Using only one (an unbounded critique-loop) is the documented failure mode that inflates scores or never terminates.
  4. Human placement and self-improvement both trend toward “gated and configurable.” The mature pattern: let the loop DRIVE autonomously but never let it ACQUIT or self-modify unilaterally — surface progress async, and require a distinct authority to approve terminal gates and land any harness self-edit. Frameworks that skip this are exactly where context-pollution and self-poisoning risks concentrate.
Section 05

Trust, Memory & Human Alignment

Strip away the domain differences and the eight frameworks converge on the same hard problem: an autonomous agent that runs unsupervised will hallucinate, drift, and declare victory it hasn't earned. Every framework answers along three axes. Verification asks “is this claim actually true?” Memory asks “what did we already learn?” Human alignment asks “where does the person stay in control?”

The most important abstraction is a single principle, stated almost verbatim by ARIS (“a loop can DRIVE; it cannot ACQUIT”) and superpowers (“Do Not Trust the Report”): the same agent that produces work must never be the sole judge of whether it is correct. Frameworks differ in how they enforce this — a different model family, a fresh context, a deterministic whitelist, an immutable metric, a human gate — but almost all enforce it somewhere. The few that don't reveal the boundary condition: you can skip the verifier only when the ground truth is so cheap and objective there is nothing to fake.

T1 The Driver-Cannot-Acquit Principle (Independent / Cross-Model Verification)

What it is. The producing agent is structurally barred from being the final judge of its own output. Verification is delegated to a different model family, a fresh context stripped of the producer's narrative, or a separate reviewer role. The shared insight: self-review is gameable — a model that wrote something will rationalize it, and a reviewer that accumulates history starts defending its earlier judgments.
  • ARIS — strongest form: mandatory cross-family heterogeneity (Claude executor / GPT-5.5 reviewer) plus a fresh thread per gate; a NeurIPS run showed thread reuse inflating scores 3/10→8/10, which codified the freshness rule. skills/shared-references/acceptance-gate.md
  • academic-research-skills — opt-in ARS_CROSS_MODEL routes integrity checks and Devil's Advocate to GPT/Gemini; divergence flagged, never averaged; motivated by a 31% same-model citation-error rate. shared/cross_model_verification.md
  • superpowers — same-model but separate-context: a spec-compliance reviewer subagent is told to distrust the implementer's report and read the actual code. skills/subagent-driven-development/spec-reviewer-prompt.md
  • Scientific Agent Skills — cross-model only for figures: one model generates, Gemini reviews against numeric quality thresholds until met. skills/scientific-schematics/SKILL.md
  • gpt-researcher — weakest form: same-model ReviewerAgent checks each section and loops to ReviserAgent; no model-family or context separation. multi_agents/agents/reviewer.py
Why it matters — This is the central anti-deception mechanism. Without it, an autonomous loop optimizes the reviewer's approval rather than the truth, and overnight runs silently converge on confident nonsense. The gradient from ARIS to gpt-researcher is the single best predictor of how much a framework can be trusted unsupervised.

T2 Fetch-Don't-Fabricate (Citation Grounding)

What it is. References and quantitative claims must be retrieved from an external authority, never generated from model weights. Implemented as a hard rule against memory-sourced BibTeX plus deterministic API verification against scholarly indices. This is the most universal pattern in the corpus — nearly every framework that writes prose enforces it.
  • AI-Research-SKILLs — cardinal rule: “NEVER generate BibTeX from memory; ALWAYS fetch programmatically” — citing a ~40% hallucination rate for AI-generated references. 20-ml-paper-writing/ml-paper-writing/SKILL.md
  • academic-research-skills — most layered: Tier-0 Semantic Scholar verification (Levenshtein ≥0.70) per citation, plus three-layer anchors enabling an opt-in claim-faithfulness audit; motivated by 146,932 hallucinated citations found in 2025. academic-pipeline/SKILL.md
  • AutoResearchClaw — 4-layer citation check (arXiv→CrossRef/DataCite→S2 title→LLM relevance) auto-removes hallucinated refs from the .bib and .tex. researchclaw/pipeline/verified_registry.py
  • Scientific Agent Skills — verify_citations.py resolves every DOI against CrossRef and flags failures; marked “CRITICAL.” skills/literature-review/SKILL.md
  • gpt-researcher — grounding-by-retrieval rather than verification: claims cite a visited URL and SourceCurator ranks credibility, but there is no API-level existence check. gpt_researcher/skills/curator.py
Why it matters — Hallucinated citations are the signature failure of LLM research output and the most dangerous one. The near-total convergence makes the divergence telling: frameworks targeting publishable papers verify existence against a registry, while the web engine only grounds against whatever it scraped — adequate for a report, insufficient for a bibliography.

T3 Reject-Before-Write Gates (Deterministic Faithfulness Guards)

What it is. A non-LLM, often pure-Python guard sits between generation and persistence: if the output violates an invariant, no bytes hit disk. Unlike a reviewer (which judges quality), these gates enforce mechanical contracts — a number must be in a ground-truth whitelist, a write must stay inside a declared directory, a success claim must be backed by command output in the same message. Deterministic, un-gameable by prompt, fail closed.
  • AutoResearchClaw — VerifiedRegistry builds a whitelist of every ground-truth number (plus rounding/percentage variants); paper_verifier rejects/sanitizes any unverified number in Results sections. researchclaw/pipeline/paper_verifier.py
  • superpowers — iron law: no completion claim without running the verification command in the same message and reading output; “Claiming work complete without verification is dishonesty, not efficiency.” skills/verification-before-completion/SKILL.md
  • academic-research-skills — PreToolUse hook deterministically denies Write/Edit outside a Bucket-A agent's phase dir and denies Bash wholesale — “all-deny is the only policy that reaches ZERO fail-open by construction.” scripts/ars_write_scope_guard.py
  • autoresearch — minimalist version: evaluate_bpb() is an immutable oracle marked “DO NOT CHANGE”; the keep/discard gate is a single objective metric the agent cannot reinterpret or forge. autoresearch/prepare.py
Why it matters — Reviewers and juries are expensive, slow, probabilistic; deterministic gates are cheap, instant, and provably fail-closed. The best frameworks layer both — a whitelist/scope guard catches mechanical cheats, freeing the LLM jury to focus on semantic judgment. This is the cleanest answer to “context pollution → deception”: the guard doesn't reason about intent, it just refuses the write.

T4 Pre-Registration & Test-First Discipline (Temporal Proof Against Post-Hoc Rationalization)

What it is. The plan, hypothesis, or test is committed before results exist, creating forensic evidence that the agent didn't reverse-engineer its reasoning to fit an outcome. In ML loops this is git pre-registration (protocol commit precedes result commit); in software loops it is TDD (watch the test fail before writing code, so the test is evidence rather than assertion).
  • AI-Research-SKILLs — hard rule: “Protocol commits MUST precede result commits. The git history is your lightweight pre-registration.” Plus mandatory inner-loop sanity checks. 0-autoresearch-skill/SKILL.md
  • superpowers — iron-law RED-GREEN-REFACTOR: watch each test fail for the right reason before writing production code; pre-existing untested code must be deleted and rewritten. skills/test-driven-development/SKILL.md
  • academic-research-skills — Sprint Contract: reviewers produce a paper-content-blind Phase-1 pre-commitment (scoring plan, failure conditions) before the paper-visible Phase 2 — the review-side analogue of pre-registration. academic-paper-reviewer/SKILL.md
  • AutoResearchClaw — weaker form: a pitfall detector scans generated code for hardcoded metrics, identical ablations, and NaN/Inf before execution, blocking fabricated-result shortcuts up front. researchclaw/pipeline/stage_impls/_execution.py
Why it matters — The subtlest autonomous failure isn't an obvious lie — it's the agent quietly fitting its story to whatever came out, which no output-inspection catches. Locking the plan in time (a git SHA, a failing test, a blind scoring contract) converts an unfalsifiable narrative into a checkable record. It is scientific pre-registration ported into the agent's version-control and test harness.

T5 Durable Project Memory with Privileged Negative Results

What it is. A persistent cross-session store — a research wiki, findings file, or lesson ledger — that the agent reads at the start of each cycle and writes back after each loop. The recurring sophistication: failed ideas and lessons are the highest-value memory — never pruned, always re-injected, used as an anti-repetition banlist. A second tier (provenance/maturity tracking) records who decided what and promotes repeated observations to durable claims.
  • ARIS — per-project wiki (paper/idea/experiment/claim entities + typed edges + 8000-char query_pack); failed ideas are most-privileged and feed /idea-creator as a banlist; a capture_filter guards against self-poisoning. skills/research-wiki/SKILL.md
  • AI-Research-SKILLs — file-based: findings.md (Current Understanding / Lessons / Open Questions) re-read every loop; the rigor reviewer penalizes exploration trees with no dead ends. 0-autoresearch-skill/templates/findings.md
  • AutoResearchClaw — EvolutionStore (JSONL) classifies stage-failure lessons into 6 categories with 30-day time-decay; build_overlay() injects relevant past lessons into each subsequent run. researchclaw/evolution.py
  • academic-research-skills — partial: the Material Passport carries cross-session artifacts/verification status, but it is a state carrier, not a learned-rule store. shared/handoff_schemas.md
Why it matters — Memory is the axis of sharpest divergence. The overnight frameworks cannot function without it — a 30-experiment loop that forgets its own dead ends will thrash. The single-session frameworks deliberately have none, accepting amnesia as the price of simplicity. Privileging negative results is the non-obvious best practice: it stops an autonomous agent from cheerfully rediscovering the same failure every night.

T6 Harness Self-Evolution (The Loop That Rewrites Its Own Rules)

What it is. An outer meta-loop observes the system's own execution (event logs, failure patterns, repeated user workflows) and proposes changes to the skill/rule definitions themselves — converting accumulated lessons into persistent instructions. Critically, mature implementations stage these changes but require a separate human act to apply them: the producer of a rule patch is never allowed to silently land it.
  • ARIS — /meta-optimize analyzes hook event logs, proposes minimal SKILL.md diffs gated by advisory review, stages under .aris/meta/pending/ — but never applies; landing requires a human /meta-apply. skills/meta-optimize/SKILL.md
  • AutoResearchClaw — MetaClaw converts high-severity lessons into persistent arc-* SKILL.md files; next run's SkillRegistry injects them — a measured +18.3% composite-robustness A/B gain. researchclaw/metaclaw_bridge/lesson_to_skill.py
  • Scientific Agent Skills — autoskill mines repeated screen workflows (PII-redacted locally), clusters them, and LLM-judges reuse/compose/novel to draft new SKILL.md proposals — staged for explicit promote.py. skills/autoskill/SKILL.md
  • superpowers — manual but TDD-driven: skills are written by running pressure scenarios without the skill (RED), authoring to address rationalizations (GREEN), closing loopholes (REFACTOR). skills/writing-skills/SKILL.md
Why it matters — This is how a framework stops repeating systemic (not just per-experiment) mistakes — the personalized self-learning layer. The shared safety discipline is the propose/stage/human-apply split, precisely because an agent that can silently rewrite its own guardrails can rewrite away its own constraints. Self-evolution without a human apply-gate would be the most dangerous capability in the corpus.

T7 The Autonomy Dial (Staged Human Gates with a Configurable Pause Surface)

What it is. Human involvement is not binary but a tunable spectrum: discrete checkpoints at high-stakes stage boundaries plus a knob controlling how often the loop pauses. The convergent design separates decision-heavy gates (the human chooses a branch) from acknowledgment gates (machine verifies, human signs off), paired with an asynchronous report surface so an absent human can review and redirect after the fact.
  • academic-research-skills — richest: Adaptive Checkpoint System with FULL/SLIM/MANDATORY types; an awareness guard forces a FULL checkpoint after 4+ consecutive continues. academic-pipeline/SKILL.md
  • AutoResearchClaw — six intervention modes plus SmartPause, dynamically triggering a pause from a weighted confidence signal (quality, LLM confidence, novelty risk, rejection rate, criticality). researchclaw/hitl/smart_pause.py
  • ARIS — AUTO_PROCEED auto-continues after 10s vs waits indefinitely; HUMAN_CHECKPOINT pauses each round; Feishu push sends scores+weaknesses — async HCI for an absent human. skills/auto-review-loop/SKILL.md
  • superpowers — front-loaded gate: a HARD-GATE blocks all implementation until the user approves the spec, then execution runs continuously with no mid-task check-ins. skills/brainstorming/SKILL.md
  • autoresearch — minimal pole: a NEVER STOP directive forbids mid-loop pausing; HCI is purely front-loaded and back-loaded — zero interaction during the run. autoresearch/program.md
Why it matters — This is the HCI-alignment axis, where intent shows most clearly. Paper-publication frameworks gate heavily because a wrong-but-confident paper is unrecoverable; ML-training loops gate almost never because a bad experiment just gets git-reset. The shared best practice is the configurable dial plus an async report surface — it lets the same framework serve a hands-on user and a sleeping one.

Takeaways

  1. Convergence is strongest on faithfulness, weakest on memory. Nearly all eight enforce fetch-don't-fabricate grounding and some producer≠judge separation; but durable cross-session memory cleanly splits the field — sophisticated in the long-horizon overnight systems, deliberately absent in the single-session ones.
  2. The trust ceiling is set by the verifier's independence. Frameworks safe to run truly unsupervised pay for cross-family or deterministic verification with a fresh context; the ones that self-review in the same model and context are best kept on a short human leash. “A loop can DRIVE but cannot ACQUIT” is the load-bearing principle of the whole space.
  3. Layering beats any single guard. The most trustworthy systems stack a cheap deterministic gate (whitelist, write-scope guard, immutable metric, failing test) that catches mechanical cheats with an expensive semantic jury (cross-model reviewer) that catches reasoning errors. autoresearch shows the boundary: drop the jury only when ground truth is a single number nobody can forge.
  4. Self-evolution must keep a human apply-gate, and gating should scale with irreversibility. The mature systems all let the system propose rule changes but require a separate human act to land them. Correspondingly, paper-publication pipelines gate heavily while training loops barely gate; the right design is a configurable autonomy dial plus an async report surface, not a fixed policy.
Section 06

Synthesis: The Canonical Auto-Research Loop

Three lenses, one machine

Read separately, the three lenses of this survey look like three different surveys. Read together, they are three views of the same machine — and that machine is more uniform than any single framework's marketing suggests.

Paradigmseven reusable functional roles — Scope, Search/Read, Ideate, Experiment, Verify, Write, Remember — the parts.
Harnessa stage-machine or a self-orchestrating skill agent that hosts those parts and isolates context — the wiring.
Workflowthe Scope→…→Report spine plus an iteration wrapper that drives, gates, and terminates — the motion.

Put them in sequence and the canonical auto-research loop falls out: a Scoper fixes a typed problem; a Searcher/Experimenter gathers ground-truth signal; an Ideator proposes the next move (consuming a memory of past failures); a Writer synthesizes a grounded artifact; a Verifier that is structurally independent of the producer accepts or rejects it; a durable Memory records the outcome — especially the dead ends; and a wrapper loop iterates this under a tick, a metric-gate, or a critique, while a configurable human dial and an async report surface keep a person in the loop without blocking it.

Everything that looks like a difference between these eight systems is a setting on this one loop: how heavy each role is, whether the harness is code or prose, whether the verifier shares the executor's model family, and where the human is wired in. The single most consequential setting — the one that most predicts whether a framework can be trusted to run while you sleep — is the independence of the verifier. Build the loop once; tune those dials per task; and never let the part that produces the work be the part that signs off on it.
