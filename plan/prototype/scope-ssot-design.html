Section 01

What it is, and its one job

The Scope SSOT is a small, passive, canonical store of intent. Its one job is to hold — at three levels and in one place — the answer to “what are we trying to do, and what counts as success?”, and to let every other component read that answer rather than re-deriving (and drifting from) it. It is the backbone that keeps the agent aligned while research churns; it is explicitly not an orchestrator and not a place results are stored.

One line to remember the whole design: the SSOT owns the yardstick (what to measure, what counts as success), never the reading (the measured value). Every other property below exists to protect that line.

Three principles fix its shape: it is canonical (every surface — plan.html, the tracker, Current Best — is a projection rendered from it, never a second copy); it is append-only and versioned (a change is a new entry with provenance, not an in-place edit); and it is written only through one gated path (the same single-mutation-surface discipline the base pipeline already enforces via research-op). Read freely; write only through a gate. And it is not built from scratch: each level already exists as a base-pipeline surface — a Direction is a research package (§7).

Section 02

The data model — typed nodes on a DAG

Scope is a graph of typed nodes at three levels. The shape is a DAG, not a strict tree: a node may be multi-homed under several parents (a baseline or task shared by two directions). It is stored once and referenced many times — edges, not copies — so there is still exactly one source of truth.

project → direction → task — the dashed red edge is a multi-homed (shared) task

Every node shares a small base; each level then adds its own yardstick fields — and only yardstick fields. Note what is absent: there is nowhere to put a result.

ScopeNode = {
  id          : string             # stable, e.g. "dir/contrastive-v2"
  level       : "project" | "direction" | "task"
  parents     : id[]               # DAG — [] for project, >=1 elsewhere (multi-homing OK)
  version     : int                # bumps on every transition that touches this node
  status      : "active" | "superseded" | "archived"
  yardstick   : Yardstick          # level-specific — INTENT ONLY, never a measured value
  provenance  : txn_id             # the transition that last changed this node
}

Reference-counted invalidation falls straight out of the DAG: a multi-homed node is invalidated only when its last owning parent drops or changes it. While any parent still needs it, it carries (worked in §6).

Section 03

The transition record — how scope changes

Scope is never edited in place. A change is an append-only transition record, and the current scope is the fold of all transitions. Each record is evidence-anchored: it says what triggered the change, why, which change-gate cleared it, and what it cost downstream. This is the causal timeline that makes “moving the goalposts” auditable instead of silent. 

Transition = {                  # append-only; scope = fold(transitions)
  txn_id, scope_version
  node_id, level
  op           : "create" | "revise" | "supersede" | "reopen" | "archive"
  delta        : field-level from -> to
  trigger      : evidence_ref     # WHAT caused it — cited artifact / experiment id
  cause        : string           # WHY — human-readable
  gate         : "user" | "user+xmodel-audit" | "agent+async-ack"   # which gate cleared it
  invalidates  : memory_id[]      # results/lessons now measured on an old yardstick
  reopens      : idea_id[]        # failed ideas viable again under the new scope
  dial_revert  : task_id[]        # tasks auto-reverted to Supervised until re-grounded
}

The gate field is set by the level of the changed node, not chosen freely — that is how “user-monitored” and “autonomous” coexist. The strength scales with irreversibility:

Table 1 — graduated gating: the node's level fixes the change-gate. Higher level = more irreversible = harder gate.
LevelChange-gate (gate)Who clears itReversibility

Directionuser ratification + cross-model evidence audithuman + L2 verifierhigh — changes the yardstick

Section 04

The non-negotiable — yardstick vs. reading

If the SSOT could hold a result, “highest priority” would let it launder a weak result into a success by editing the truth to fit — the ultimate deception engine. So authority is layered: the SSOT is the top of the normative stack (what we should do); verified runtime artifacts are the top of the empirical stack (what actually happened); and a gate sits between them, comparing the two. The SSOT supplies the bar; the tracker supplies the number; neither can overwrite the other.

Normative top vs. empirical top — the gate compares, never merges

This is also how the SSOT coexists with the base pipeline's “runtime truth wins” rule: they are different layers. The yardstick is authoritative about intent; the runtime artifact is authoritative about outcome; changing the yardstick to match an outcome is possible only as an audited transition (§3), never as a quiet write.

Section 05

Function — the API surface

The SSOT is a passive reference: many cheap reads, one gated write, and projections that render from the canonical store. It runs nothing on its own — components pull intent from it; an accepted transition is the only thing that mutates it, and that transition triggers propagation.

Table 2 — the whole surface. Reads are free; the single writer is gated; propagation fires only on an accepted write.
OperationKindGated?What it does
intent(node)readnofold the transitions into the node's current yardstick
yardstick_for_gate(task)readnothe success predicate + metric a gate compares against — gates pull their bar from here
history(node)readnothe causal transition timeline (from→to, trigger, cause)
project(surface)readnorender plan.html / tracker / Current Best — derive, never duplicate
propose_transition(txn)writeyes — the only writervalidate → change-gate (Table 1) → append → propagate. Reject-before-write.
propagate(txn)internal—fires only on an accepted txn: carry / invalidate / reopen + dial-revert

Two function rules that enforce the design (not instruct it)

  • One writer, and it is a research-op gate entry. propose_transition is registered in the existing gate matrix as a reject-before-write op; an illegal transition (wrong gate for the level, a reading written into a yardstick field) never hits disk.
  • Gates pull, they are not told. A verifier reads yardstick_for_gate at the moment of judging, so it cannot act under a stale scope — the priority is enforced mechanically, not declared in a prompt.
Section 06

Expected usage — three walkthroughs

A. Read at a gate — the yardstick is enforced

The Verify role reaches an acquit gate on Task T. The gate does not trust the agent's claim; it pulls the bar and compares.

1Hit acquit gateTask T wants status → success
2Pull yardstickyardstick_for_gate(T): “nDCG@10 ≥ baseline + 2”
3Read runtimetracker says +0.5 (verified artifact)
4Compare → REJECTpredicate false; stays in-progress

The SSOT never saw the number; it only supplied the bar. The reading came from the runtime artifact. The gate between them produced the verdict — layered authority doing its job.

B. A scope change — write path with propagation

An experiment shows the chosen metric has saturated; the direction needs a better metric. This rewrites a yardstick, so it is a direction-level transition — the hardest gate short of a project pivot.

Transition record (appended on user ratification)
oprevise — node dir/contrastive-v2 (direction)
deltametric: Recall@10 → nDCG@10; predicate updated
triggerexp #42 — Recall@10 plateaued across 3 configs (cited artifact)
causemetric no longer discriminates ranking quality
gateuser+xmodel-audit — blocked for the user; L2 audited the evidence
invalidates3 results + 1 “Current Best” measured on Recall@10 → archived
reopensidea tiling-attn — failed only on Recall@10, viable on nDCG
dial_revert2 tasks under this direction → Supervised until re-grounded

On acceptance, propagate runs the carry / invalidate / reopen pass over memory so nothing stale survives silently:

Carryunaffected by the new metric — stays active
Invalidatemeasured on Recall@10 — archived, not deleted (pointer kept)
Reopena failed idea now viable under nDCG — unbanned

C. Multi-homing — reference-counted invalidation

A shared baseline node B is referenced by both Direction A and Direction B.

1Supersede Dir AA drops its reference to B
2B survivesDir B still owns B — refcount > 0, carry
3Later: supersede Dir Blast owner gone — refcount = 0
4B archivedonly now, and as a recorded txn

This is why the hierarchy is a DAG: invalidation follows references, not position, so a shared asset is never dropped while someone still depends on it.

Section 07

How it maps onto the base pipeline — Direction ≈ a research package

The SSOT is not built from scratch. Its three levels already exist as base-pipeline surfaces, so most of the Direction level is reused, not net-new — the base pipeline literally scaffolds packages per direction. The SSOT's job is to formalize, version, and connect what those surfaces already hold.

Table 3 — each SSOT level is an existing base-pipeline surface; the SSOT reuses its intent fields and adds the spine around them.
SSOT levelBase-pipeline surfaceReused as the yardstickWhat the SSOT adds
Projectresearch-dashboard + RESEARCH_PROJECT_PROFILE / contribution spinenorth-star · spine · non-goalsone canonical, versioned home (vs. schema.js + a derived learnings view)
Directiona research package (its plan.html intent)hypothesis · metric · baselines · gate · laneyardstick/reading split enforced; versioned transitions for yardstick changes
Taskan experiments[] pipeline-timeline phase (P0, P1, …)purpose · after · output · gate · statusautonomy_level (the dial); multi-homing across directions

The correction that matters: a Direction node is the package's yardstick slice

A research package holds the reading as well as the yardstick — results.html, tracker live state, methodsTried (measured / verdict), Current Best. The SSOT Direction node must hold none of that (§4). So a Direction is not the whole package; it is its intent slice.

One research package = the SSOT Direction node (yardstick) + the empirical layer the SSOT must NOT own

→ Scope SSOT Direction node — normative
plan.html intent: hypothesis · metric definition · baselines · success predicate · gate · lane status
stays empirical — NOT in the SSOT
results.html · tracker live state · methodsTried (measured / verdict) · Current Best

The result-gate rows on results.html already sit between the two halves — that is §4's gate, latent in the package today.

What the SSOT adds that the package model lacks

  • Versioned transitions for yardstick changes. A package edits its identity fields in place (lastUpdated toggles staleness); the SSOT records each yardstick change as an append-only, evidence-anchored, gated transition (§3) — closing the silent-goalpost-move gap.
  • The autonomy dial. autonomy_level is a new Task field; the package's experiments[] entry has no equivalent (built in build-order Stage 2c).
  • A DAG + one cross-level backbone. Packages are independent units and learnings.html is only a derived read; the SSOT unifies project → direction → task into one versioned spine and lets a task multi-home across directions (§2).
  • A clean status split. The package's (category, status) conflates direction lifecycle (brainstorm → success / fail) with task execution-state (EXPERIMENT_RUNNING, …); the SSOT splits these across the Direction node (active / superseded / archived) and the Task node.
One open design fork (resolve before Stage 0 code): does the SSOT sit above research-packages.js, making the inventory itself a projection of the canonical store — or does it reuse the inventory as its store and add only the versioned-transition log + project/task structure on top? The “canonical store, surfaces are projections” principle (§5) points to the former; the existing pipeline treats research-packages.js as canonical and learnings.html as derived.
Section 08

Stage 0 build scope + the TDD gate

Stage 0 builds the thinnest version of all the above that still enforces the invariants — schema + the one gated writer + propagation. Behaviour (real roles, the verifier, the dial) comes in later stages; here we only lock the contract and prove it rejects.

Build now — defer the rest

  • Build: the ScopeNode / Transition schema; propose_transition as a research-op gate entry with Table 1's gate logic; the carry/invalidate/reopen + dial-revert propagation; read accessors (intent, yardstick_for_gate, history).
  • Defer: one project at a time (no portfolio); rich projection rendering (a minimal project() is enough); sophisticated reopen heuristics (start with exact metric-match); multi-homing supported in the schema but exercised mainly at task level.

The stage is done when these were red first, then green:

TDD gate — Scope SSOT

  1. Schema round-trips a fixture (parse → serialise → equal); an out-of-schema field is rejected.
  2. A reading written into a yardstick field is REJECTED — the non-negotiable, enforced at the boundary (§4).
  3. A direction-level propose_transition without gate = "user+xmodel-audit" is REJECTED (Table 1).
  4. Propagation is correct: a metric-revise marks results measured on the old metric as invalidates and reopens exactly the ideas that failed only on it (fixture-checked).
  5. Reference-counting holds: a multi-homed node is not invalidated while any parent is still active (walkthrough C).

Tests 1 and 3 are the two seeds the build-order plan names as the very first step. The other three lock the invariants this page exists to protect.
