Section 01

The harness in one picture

The base pipeline (Trustworthy-Research-Pipeline/) is not a full auto-researcher. It is a complete, tested trust + execution substrate for the experiment loop of a single, already-scoped research direction. Read it as the bottom half of the survey's canonical machine: it owns Experiment, the deterministic half of Verify, the HCI surface, the audit trail, and durable research-content Memory — and it deliberately leaves the autonomous front end (Scope / Search / Ideate), cross-model independence, and harness self-learning unbuilt.

Everything in Sections 2–5 is one of these four guarantees seen from a different angle. Sections 6–7 then overlay the survey's canonical loop to expose the gap.

Section 03

The state machine — four lanes

Every package carries a (category, status) pair declared in schema.js. The category is the lane; the status is the position within it. This pair drives the required-field rules, the legality matrix, and the learnings lint. Crossing a lane boundary is a terminal transition and requires user acknowledgement (rule T1).

Section 04

The seven-step workflow loop

WORKFLOW.md is the decision-owner controller the agent runs inside one in-progress package. The agent holds global context and judgement; bounded subagents supply evidence, implementation, review, monitoring, and analysis but are never the final authority. The goal is the hypothesis verdict in results.html + next-action.html — not landed scaffolding. The loop already drives autonomously through these steps, scheduling its own re-entry while runs are open.

Iterate — the loop's back-edges
  • Step 3 → Step 2 — reviewer returns blocking findings → consolidated fix brief (reject before launch).
  • Step 5 ↔ Step 4 — the 10-minute live cycle; continue keeps the run, armed re-entry via ScheduleWakeup / Monitor.
  • Step 5 → Step 2 — a concrete code/function issue surfaces mid-run → back to implement.
  • Step 7 → {Step 4 · Step 2 · BLOCKED · STOPPED} — config/seed/ablation follow-up · code change · missing user decision · goal reached.

The gates riding on this loop (all built)

  • Mutation Rule — every surface edit in every step goes through research-op (§5).
  • Fact Propagation Contract — each landed artifact (checkpoint, candidate JSON, sentinel, phase marker, chain-done) is fanned out to every owning surface in the same turn via scan-events → event.
  • Stop Gate — the turn may only end in BLOCKED/STOPPED, with scan-events empty and a re-entry armed if any run is open. This is the “never idle while work is open” tick — but scoped to one package's experiment loop.
  • T1 terminal-ack — a lane-crossing status change pauses for the user. This is the existing “DRIVE but do not ACQUIT” seam.

Read the colours: Steps 1, 2, 4, 5, 6, 7 are have — fully drive-autonomous. Step 3 is partial — it dispatches multiple reviewers, but they are same-model, separate-context subagents; there is no cross-family judge (§7, the highest-leverage gap).

Section 05

The single mutation surface & the five contracts

Direct Edit/Write on a package file is a workflow violation. Every mutation funnels through research-op, which validates against the (category, status, op, target) legality matrix and per-target invariants before any byte hits disk (Pattern B, reject-before-write). Every attempt — accepted or rejected — appends one line to the audit log.

Reject-before-write funnel

insert methodsTried / result-gate row update status / activeGate / metric delete doc-file / row event chain-done / checkpoint

✓ legalwrite surface + re-paint HTML + 1 line to _actions.jsonl
✗ illegalno bytes written; structured rule-violation envelope returned to the agent to retry

The matrix is concrete — e.g. a methodsTried[] row is only insertable in (in-progress, RESULT_ANALYSIS / NEXT_ACTION_READY) and is frozen (delete-forbidden) once a package reaches success/fail; a lane-crossing status update requires a T1 ack. This is the mechanical core of problem #1: deception is structurally cheaper to avoid than to commit, because the write simply fails.

It is not harness-level self-learning (reflection over the agent's own behaviour → rule changes). That distinction is the whole of problem #3 and is taken up in §7.

Section 06

Coverage map — harness vs. the canonical auto-research loop

Now overlay the survey's seven functional roles. The harness covers the downstream, empirical half of the loop densely and the upstream, generative half barely. The single chain below is the clearest statement of what you have and what you must build.

How to read this chart

  1. The harness is the bottom-right of the survey's loop. R4 (Experiment) is its centre of gravity and is fully built; R5's deterministic half (metric-gate, reject-before-write) is built too. That is exactly the “autonomous scientist with an unfakeable metric” branch the survey praises.
  2. The upstream is open. R1–R3 (turn intent into a scoped, hypothesised, prioritised next move) run in your head today. Automating them is most of the “auto” in auto-research.
  3. Two PARTIALs are really trust gaps, not coverage gaps. R5 (cross-model independence) and R7 (self-learning) are present in weak form; upgrading them is where problems #1 and #3 are actually closed.
Section 07

What is missing — the gap analysis

Six concrete gaps, each tied to a core problem from 核心问题.md and to its survey anchor. The first three are new roles; the last three are harness settings — and the harness settings are where the trust actually lives.

Table 2 — the gap from current harness to autonomous trustworthy pipeline.
Gap核心问题Current stateWhat to addSurvey anchor
Autonomous Scope / Search / Ideate missing #1 alignment workflow begins at “a mature research plan”; the front end is human upstream roles that emit a typed problem + ranked candidate move into plan.html/brainstorm R1–R3; canonical Scope→Plan→Gather
Cross-model / cross-family verifier missing #1 deception multi-reviewer but same-model, separate-context; no independent judge a fresh, different-family reviewer on the acquit gates; producer ≠ judge enforced T1 “DRIVE but cannot ACQUIT” — best predictor of unsupervised trust
Whole-pipeline outer loop partial #1 + #2 the loop drives one package's experiment cycle only; choosing the next direction is human a tick/continuity loop over directions + an async report surface so an absent user can redirect W4 iteration mechanism; W6/T7 autonomy dial + async report
Self-learning / self-evolution missing #3 durable content memory only (learnings.html, methodsTried) a reflection loop over _actions.jsonl + transcripts → proposes rule/skill diffs, human apply-gate T6 / W7 — propose-but-gate-landing (never auto-apply)
User-led Task & Objective surface partial #2 plan.html objective + contribution spine exist, but no explicit user→agent task/objective alignment surface a typed objective/task channel the user steers and the loop aligns to each cycle #2 solution 2 (user-led Task/Objective system)
Fetch-don't-fabricate citation grounding missing #1 faithfulness not present only if literature/report output enters scope; the experiment branch leans on the metric oracle instead T2 — conditional on the experiment-vs-document fork

The one that matters most

  • If the pipeline is meant to run while you sleep, the cross-model verifier is the highest-leverage single addition: the survey calls verifier independence the best predictor of unsupervised trust, and same-model self-review “silently converges on confident nonsense.” Your harness already has the cheap-rigid half (the metric-gate is your evaluate_bpb()); the move is to layer an independent critic on the acquit gates, not to replace what works.
  • Self-learning (#3) is necessarily last: it consumes the audit log and memory the rest of the harness emits, so it cannot be built before them.
Section 08

What this implies for build order

The dependency order falls straight out of the map: the trust substrate already exists, so the work is a short staged extension — contracts → a thin end-to-end skeleton → deepen roles (verifier first) → self-learning last. This is the on-ramp; the detailed build-order plan is the next document.

Dependency stages — colour = readiness

mostly doneStage 0

Contracts & seams

The state machine, gate matrix, audit log, dashboard. Confirm the new roles' typed interface fits the existing matrix; add only the seams they need.

you are hereStage 1

Walking skeleton

One topic → run the entire loop end-to-end through the real gates, each new upstream role a stub. Prove wiring + dashboard + audit before any role is deep.

Stage 2

Deepen roles — verifier first

Cross-model verifier on the acquit gates, then auto-Ideator with a failed-idea banlist, then Scope/Search as the domain needs. Each TDD, each gated.

Stage 3

Self-learning outer loop

Reflection over _actions.jsonl + transcripts → proposes rule/skill diffs → human apply-gate, never auto-apply. Closes problem #3.

Why this order, in one line each

  1. Stage 0 before all — every role validates against the gate contract; it is the spec they are written to (and it is already built).
  2. Stage 1 before Stage 2 — prove the harness end-to-end with stubs before spending effort deepening any single role; a tracer bullet, not a big bang. Avoids the survey's “more context made it measurably worse” trap.
  3. Verifier first within Stage 2 — it is the trust ceiling; building generative roles on top of a weak judge just produces confident nonsense faster.
  4. Self-learning last — it can only reflect on signals the working harness emits.
