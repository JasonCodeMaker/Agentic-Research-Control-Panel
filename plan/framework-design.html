Detailed design — usage-first, one system

Framework Design — The Trustworthy Auto-Research Pipeline

The detailed-design layer between the north-star (plan/target-architecture.html) and implementation. It is organised the way you use it: a single seven-step journey is the spine, and every plane, skill, library, and gate is introduced as the machinery one step of that journey actuates. One system — no base/auto split, one mutation surface — with every hard constraint enforced by a gate, a lib check, or a privilege split, never a prompt.

Section 01

How you use it — the seven-step journey

This is the spine of the whole design, and everything else hangs off it. The pipeline is a management interface: you are the project manager, the agent is a managed worker whose tasks, state, evidence, and next actions are made visible and governable (this is Solve #2 made literal). The seven steps below are your workflow; every component in §3–§5 exists only to serve one of them. That is also the complexity filter the spec demands — if a skill, lib, or gate can’t be traced to a step you actually perform, it doesn’t get built.

Section 02

The machine the journey runs on — four planes + the Scope SSOT backbone

The journey runs on the north-star’s machine: four planes, one concern each, threaded by the Scope SSOT backbone. The control plane is where you steer; the execution plane does the research; the trust plane gates what is allowed and what is true; the memory plane is what compounds. The planes are a conceptual cross-section — which concern a unit serves — not directories. Physically it is one catalog.

One catalog, not two trees

Because the design is scoped to this repo and the base is reused (and, where a journey step needs it, extended) in place, the whole machine lives under one root as one peer catalog of skills + libraries + shared contracts. There is no auto/ sibling and no import wall.

What the one-system reframe deletes. The earlier two-tree draft’s auto/ sibling tree, Seam-A / Seam-B registration plumbing, test_dependency_law.py, the auto-namespaced-token grep, and the duplicate CLAUDE.md / WORKFLOW.md are all removed. The dependency direction is still auto-capability→base-mechanism, but it needs no test to police it, because there is no second tree and no base-edit introduced solely to cross a border. This is the single biggest cut against the spec’s minimize-complexity mandate.
Section 03

Step by step — the machinery each step actuates

The heart of the design. Each journey step is taken in turn, and the components it needs are introduced as its extension — the skill(s) it invokes, the lib(s)/gate(s) it hits, the SSOT level it touches, and the core problem it discharges. Reused base mechanism is marked reused, widened pieces extend, net-new units new.

Step 1 · Setup Control problem #2

Setup — stand up the dashboard

research-dashboard reused scaffolds research_html/ (the four lanes), the binding HTML rules R1–R17 and Trust rules T1–T24, and the project-local Stop hook. The single widening for the auto layer: the Stop hook gains scope_ssot.assert_consistent extend next to the existing propagate_apply + learnings_lint, so the dashboard’s rendered intent can never silently drift from the SSOT (invariant 3). The model→user half of HCI alignment is this dashboard.

Step 2 · Define Project Backbone SSOT: Project

Define Project — the SSOT root and its non-negotiable

research-scope new proposes the Project node; you ratify it. The write is a scope op through research-op — the same single mutation surface that gates every package edit (see §3.4) — so it is reject-before-write and audited like any other op. The SSOT is hierarchical, and the level of a change sets the gate strength: this is what reconciles “user-monitored” with “autonomous”.

Three SSOT levels — a change at one level invalidates only its dependents, and its level sets the gate

Project (global)
north-star objective · contribution spine · non-goals
change gate: mandatory user ratification — maximal irreversibility
Direction (regime)
hypothesis · metric · baselines · success predicate
change gate: user ratification + cross-model evidence audit
Task
active experiment · config · gate · autonomy level
change gate: agent may update + async notify (light ack)

The non-negotiable: the SSOT owns the yardstick, never the reading. It defines what to measure and what counts as success; it must never hold or override a measured value — or it becomes the ultimate deception engine (edit the truth to fit the result). Six invariants keep it a trust anchor rather than a new failure mode:

Table 1 — drop any invariant and the SSOT degrades into a deception engine, a complexity bomb, a second source of truth, or supervised-always. Each is enforced at a coded point (§5, Table 7).
InvariantFailure mode if dropped
1. Owns the yardstick, never the reading“highest priority” would launder weak results — catastrophic deception
2. Versioned, not a mutable snapshotin-place edits lose history and enable silent goalpost-moving
3. Canonical; surfaces are projectionsa second copy that drifts is context pollution; surfaces render from it
4. Passive reference, not orchestratora god-module that runs things bloats and fights minimize-complexity
5. Priority enforced in the gates, not declaredgates pull the yardstick from it, so the agent can’t act under a stale scope
6. Graduated gating by levelglobal = hard human gate, task = async notify; else user-monitoring kills autonomy
Step 3 · Create Direction + Task Control SSOT: Direction / Task

Create Direction + Task — and pick the autonomy dial

research-scope scaffolds the Direction as a research-package (the reused research-package reused skill — a Direction ≈ a package). You set the Task objective and pick the autonomy level. The Task/Objective surface is the Task level of the SSOT: the user→model half of HCI alignment.

The 4-stop autonomy dial — set per Task, never globally

The dial’s substance is a coupling: more autonomy automatically buys more verifier independence, so throughput and trust scale together, never against each other. That coupling has one source of truth — the autonomy_level → required-independence table is a single constant owned by the verifier lib (§3.4). research-auto passes only the autonomy_level read from the Task node; it never re-implements the tiers.

Step 4 · Run Execution + Trust problem #1

Run — the two-loop execution, gated by the trust substrate

research-auto new is the only normal front door. It drives one Task through the seven roles at the dialed autonomy in a two-loop shape: the inner loop is the base pipeline’s experiment sub-cycle, reused verbatim as R4 (it must not be re-implemented); the outer loop is research-auto ticking one Task through R1→R7, then on to the next Direction (DEEPEN / BROADEN / PIVOT / CONCLUDE — where a PIVOT is a scope-change Triage item, never a silent reroute). For absent-user operation the tick is wall-clock (/loop or Cron), and at each tick it reconstructs state by reading external memory — holding nothing load-bearing across a tick.

The single mutation surface — where the old “seam” used to be

Every write in the run — package content and scope transitions — routes through research-op’s reject-before-write gate extend. Scope/intent writes are simply a new op category in the existing (category, status, op, target) matrix; scope_ssot is the passive store the gate writes into. There is no registration hook and no parallel gate: “single mutation surface” is now literally true across both domains, and the prior Seam-A/Seam-B fork is closed.

The verifier, layered: DRIVE, but never ACQUIT

The deliverable is idea→paper, so the verifier covers both branches: the experiment branch has an unfakeable metric oracle; the document branch has no oracle and leans on independent critique + citation existence. They are layered — a cheap deterministic gate catches mechanical cheats, freeing an expensive cross-model jury to judge meaning — and the producer never signs off its own work.

Two enforced layers + the human layer the dial chooses — one principle: producer ≠ judge

reusedL1Deterministicreject-before-write · metric oracle · cite-exists
newL2Cross-model juryfresh, different-family judge on acquit gates
extendL3Humanfinal acquit at the dial’s chosen gates (T1)

The verifier lib new spawns a fresh, different-family judge, passes it file paths only (never shared context), and checks judge-identity ≠ producer-identity — the producing role has no acquit write path. The cite_check lib new resolves every citation against external indexes (arXiv→CrossRef→Semantic Scholar→optional relevance) and rejects an unresolved citation before write — a network fact, never a model’s memory. L2 is the single highest-leverage new piece, and it is exactly what the dial escalates.

Step 5 · Away modes Control problem #2

Away modes — the agent owes you context

At Async / Autonomous the agent owes context upward. Each async push is a PACK’d narrative for an absent reader — a typed bundle of {attempted, found, hypothesis-state, next-action, blocking-decision} — written via research-op so it lands on the right package surface and appends one audit line. You UNPACK by reading the dashboard, not the raw _actions.jsonl. The surfaces are a Common Information Space, not a store you must mine. Three constraints keep the PM interface honest:

Three constraints that keep the interface honest

  1. DRIVE, but cannot ACQUIT. The agent may not declare its own gated work successful (that is L2/L3) and may not edit the objective/yardstick. Trust comes from the verifier at the gate, never from the PM–worker relationship’s implied trust.
  2. Scope changes enter through the admission gate, not the worker’s pen. The agent proposes a revision as a Triage item; the human disposes; the objective cascade stays PM-write-only (§3.6).
  3. The agent owes context upward. It actively PACKs evidence so the returning human can cheaply UNPACK and re-ground — the dashboard is that narrative, not a compliance trail.
Step 6 · Scope change Backbone + Control non-stationarity

Scope change — the goalposts move, auditably

Research is non-stationary; the objective will move. research-scope files the change as a Triage item (agent proposes); you dispose. On ratification, scope_ssot records a versioned, evidence-anchored transition (from→to / trigger / cause / delta / invalidates / reopens) — the one place where moving the goalposts becomes auditable rather than silent — and runs a propagation pass over dependent memory:

·Carryunaffected by the change — stays active
·Invalidatemeasured on an old yardstick — archived, not deleted
·Reopena failed idea may be viable under the new scope

The same transition propagates to the dial: a Direction- or Project-level change auto-reverts every affected Task toward Supervised and locks it there until the agent re-grounds (read the new version, acknowledge the changed yardstick, emit a re-alignment artifact). Autonomy is earned against a shared understanding; when the purpose shifts, that grounding is stale, so standing autonomy is withdrawn until it is rebuilt. Because the SSOT is a DAG, a node multi-homed under several Directions is invalidated only when its last owning Direction drops it (reference-counted).

Step 7 · Self-learning Memory problem #3

Self-learning — the project gets better at itself, human-gated

The outer loop that makes the harness improve. It reads what the rest of the machine emits, distils durable lessons, and proposes changes to the harness’s own rules — but never lands them without a human gate.

reusedS1Observe_actions.jsonl + transcripts + transitions
newS2Reflectdoom-loops, repeated failures, scope thrash
newS3Proposestaged rule/skill diffs → pending/
extendS4Human landresearch-apply; jury-at-landing

The load-bearing defence is a structural privilege split: research-reflect new is a read-only proposer with no write tool on the live corpus — it can only stage diffs to pending/. research-apply new is the only unit with corpus-write privilege; it requires an explicit human act, runs a fresh jury-at-landing via the verifier, and stamps provenance. Two skills, two tool-permission sets — harder to erode under context pressure than a subcommand flag. Every memory entry is stamped with the SSOT version that produced it, so the failed-idea banlist is scope-conditional, not absolute, and recurring scope-thrash is itself a doom-loop the reflector surfaces. Built last, because it consumes what the other steps emit.

Why three of them are libraries, not skills

scope_ssot, verifier, and cite_check expose plain Python functions — no LLM, no slash command, no model-invocation. A skill is a prompt the model reads and may rationalize around; a lib is code that runs the same way every time. Putting the load-bearing checks (the yardstick guard, producer≠judge, the independence table, citation existence) in libs is what makes “enforce, don’t instruct” real rather than aspirational. The libs are passive — they hold no scheduler and call no Skill tool, so they structurally cannot become a god-module (SSOT invariant 4).

What this hands to the build-order plan. A closed traceability set where every function has one home, every invariant one enforcement point, and every unit one journey step — so Stage 0 is “type the scope_ssot schema + add the scope-op rows to the research-op matrix, TDD-gated” with no ambiguity about where code goes and no border to police. The acceptance test for every stage is simply: the existing base suite still passes, and every new unit is reachable from a journey step.

Provenance

The four-plane machine, the Scope SSOT backbone and its six invariants, the layered verifier, and the per-task autonomy dial all trace to plan/target-architecture.html; every subsystem traces to a problem in 核心问题.md.
