Build-Order Plan · how we get there

From north-star to a working machine — what to build first, and how to know it works

The target architecture is the blueprint: what the machine is. This is the construction schedule: the order we build the five net-new regions, and the test that proves each stage before the next begins. The spine is harness-first — a thin idea→paper walking skeleton through the real gates first, then deepen in leverage order. Every stage's definition-of-done is a passing test, never a vibe.

Section 01

How to read this plan — the build philosophy

The architecture names five net-new regions on a working substrate: the Scope SSOT backbone, the upstream roles (Scope / Search·Read / Ideate / Write), the L2 cross-model verifier + citation gate, the control plane / autonomy dial, and the self-learning loop. The wrong way to build them is one-at-a-time to completion (functions-first) — you only discover the seams don't fit at the end. So we build the other way: a thin vertical slice through the whole loop first, then deepen region by region. Every stage runs end-to-end through the real gates; what grows across stages is depth, not coverage.

Five rules that constrain the order

  1. Walking skeleton first. A thin idea→paper loop that actually runs beats any deep-but-isolated component. It retires the one risk that kills integrations — that the pieces don't compose — while it is still cheap to fix.
  2. TDD is mandatory. A stage is “done” only when a test that was red turns green. This is 核心问题 #1's own directive (all implementation must be test-driven) treated as a trust mechanism, not a nicety.
  3. Deepen in leverage order. Highest-risk / highest-leverage first: the L2 cross-model verifier before the autonomy dial that leans on it, and the self-learning loop last because it consumes what the other stages emit.
  4. Preserve the trust contracts. Every new gate is designed as a research-op reject-before-write hook that reads its yardstick from the Scope SSOT — we extend the existing gate matrix and state machine, never weaken them.
  5. Minimize complexity. Thin SSOT, one mutation surface, stub before heavy. Anything no core problem asks for is deferred (non-goals in §7).
Section 02

The sequence at a glance — and two non-obvious ordering calls

Four stages. Stage 1 is a thin slice through everything; Stages 2–3 deepen it. The depth grows; the end-to-end coverage is there from Stage 1 on.

The four stages — each runs the full loop; only the depth changes

Stage 0Contracts & seamstype the SSOT + the verifier gate hook
Stage 1Walking skeletonthin idea→paper through real gates
Stage 2Deepenverifier → roles → dial
Stage 3Self-learningreflect over what the rest emit

Inside Stage 2, deepen in leverage order: 2a L2 cross-model verifier → 2b heavy roles → 2c control plane / autonomy dial.

The two ordering calls that aren't obvious

  • Verifier before the dial. The autonomy dial's whole substance is that it escalates required verifier independence as a Task is given more freedom. You cannot safely enable the upper rungs (Async, Autonomous) before the independent verifier they depend on exists — so L2 (2a) is built before the dial (2c).
  • Self-learning last. The reflection loop reads the audit log, transcripts, and scope transitions that the other stages produce. Build it earlier and there is nothing to learn from; build it last and it has a full corpus on day one.

Section 03

Stage 0 — Contracts & seams

Before any behaviour, lock the typed interfaces everything else composes against. This is structured interfaces as a trust mechanism: if the seams are typed and the illegal move is rejected at the boundary, later stages cannot quietly violate them. Mostly a contract-and-test stage, little runtime.

Stage 0Contracts & seamsprereq: base pipeline

Goal: a stable typed surface for the Scope SSOT and the verifier gate hook, so every later stage builds against contracts, not guesses.

region: Scope SSOT核心问题 #1 · #2 · #3
Build
  • newScope SSOT schema. Typed nodes for project / direction / task, and the versioned transition record (from→to, trigger, cause, delta, invalidates / reopens). Yardstick fields only — never a measured value.
  • newTask node fields for the Task/Objective surface and the per-task autonomy_level — the seam Stage 2c fills.
  • newVerifier gate hook in the research-op gate matrix: an acquit transition must carry a verdict record; the producer identity is stamped so a judge can later be checked against it.
  • reusedConfirm the inherited seams — gate matrix, (category, status) state machine, _actions.jsonl audit log, dashboard schema.js — are read, not re-implemented.
Verify — the red tests to write first

TDD gate

  1. A fixture SSOT round-trips: parse → serialise → equal; an out-of-schema field is rejected.
  2. An acquit op submitted without a verdict record is REJECTED by the gate matrix (reject-before-write) — the verifier seam exists before any verifier does.
  3. A write to a node above the current Task's level without its required change-gate is REJECTED (graduated gating, in the type system).
Unlocks
Every later stage has a stable, enforcing surface to build against — and the two seams the whole machine hangs on (the SSOT and the acquit gate) reject illegal moves before a single role is written.
Section 04

Stage 1 — The walking skeleton

One Task on the SSOT drives a thin idea→paper loop through all seven roles and the real gates, at a single autonomy level (Supervised). Each role is a stub or thin implementation; what is real is the wiring — the gates fire, the yardstick is read from the SSOT, the audit log records the chain, the dashboard renders it. This is the stage that retires integration risk.

Stage 1Walking skeletonprereq: Stage 0

Goal: prove the loop composes end-to-end before deepening anything — a fixed intent in, a grounded skeleton paper out, every gate exercised.

regions: all 5, thin核心问题 #1 · #2
Build — thin role, real gate
  • newR1 Scope (thin). User intent → typed problem / RQ / non-goals written into the SSOT project + direction nodes.
  • newR2 Search/Read (stub) + L1 cite-exists. A couple of seed sources, but the citation-existence check is real from day one.
  • newR3 Ideate (stub). Pick one hypothesis; write it to the SSOT direction node.
  • reusedR4 Experiment. The base pipeline's loop, run on a toy task — the real metric oracle, unmodified.
  • extendR5 Verify (L1 only). Metric gate + cite-exists; the L2 slot is a placeholder that defers to the human (valid at Supervised).
  • newR6 Write (thin). A grounded-only IMRAD skeleton — claims pulled only from verified artifacts.
  • reusedR7 Remember + extend dashboard render + T1 terminal-acks at the Supervised gates.
Verify — the red tests to write first

TDD gate

  1. End-to-end smoke: a fixed intent drives all seven roles; a paper artifact is produced; the audit log shows the full transition chain, each terminal step carrying a T1 ack and reading the SSOT yardstick.
  2. Planted fabricated citation → L1 cite-exists REJECTS it; it never reaches the paper.
  3. Planted metric-gate miss → the terminal “success” transition is BLOCKED.
Unlocks
Integration risk is retired. From here, deepening any one role is a local change against proven seams — not a gamble on whether the machine fits together.
Section 05

Stage 2 — Deepen, verifier first

With the loop proven, thicken it in leverage order. 2a raises the trust ceiling (the L2 cross-model jury — the single highest-leverage gap); 2b makes all seven roles heavy; 2c turns on the per-task autonomy dial that the verifier now makes safe.

Stage 2Deepen the sliceprereq: Stage 1

Goal: go from skeleton to a heavy, trustworthy, user-steerable idea→paper loop — building the verifier before the dial that leans on it.

regions: verifier, roles, control plane核心问题 #1 · #2
2a · L2 cross-model jury — the trust ceiling
Build
  • newIndependent judge on acquit gates: a fresh, different-family model judges what L1 cannot (substantive correctness, soundness), replacing the Stage-1 placeholder.
  • newDial coupling (read-side): the verdict requirement reads the Task's autonomy_level — higher autonomy demands stronger independence.
Verify — red first

TDD gate

  1. Producer ≠ judge: an acquit where the judge identity equals the producer identity is REJECTED (“DRIVE but cannot ACQUIT”).
  2. A planted passes-L1-but-substantively-wrong claim is refuted by the jury and its acquit is blocked.
  3. autonomy_level = Autonomous without a cross-family judge refuses to start.
Unlocks: the trust ceiling rises — the dial's upper rungs become safe to enable in 2c.
2b · Heavy roles
Build
  • newR2 real Search/Read: multi-source retrieval + the full citation stack (fetch-don't-fabricate).
  • newR3 real Ideate: diverge–converge, consulting the failed-idea banlist (scope-conditional, per §2).
  • newR6 real Writer: full IMRAD with a review↔revise loop; new richer R1 scope typing.
Verify — red first

TDD gate

  1. R2: every cited claim resolves to a fetched source — no orphan citations.
  2. R3: a banlisted failed idea is not re-proposed; and it is reopened after a scope transition that invalidates the ban.
  3. R6: every paper claim id maps to a verified-artifact id (grounded-only).
Unlocks: a genuine idea→paper run on all seven roles — no longer a skeleton.
2c · Control plane & autonomy dial
Build
  • newDial upper rungs (Checkpoints / Async / Autonomous) + the per-Task selector; the Task/Objective surface as a projection of the SSOT task node.
  • newCoupling made mechanical: each dial level enforces its row of human-gates × verifier-independence × report-cadence.
  • newDial auto-revert: a direction/project scope transition reverts affected Tasks to Supervised until the agent re-grounds (per §2).
Verify — red first

TDD gate

  1. Each dial row is enforced: Checkpoints pauses at terminal gates only; Async never blocks; Autonomous requires the cross-family judge.
  2. A direction-level scope transition auto-reverts affected Tasks' dial to Supervised.
  3. A scope change the agent proposes lands as a Triage item the human disposes — the objective cascade stays PM-write-only.
Unlocks: per-task configurable autonomy is live; HCI alignment (#2) is complete in both directions.
Section 06

Stage 3 — The self-learning loop

Last, because it consumes what every other stage emits. The loop observes the audit log, transcripts, and scope transitions; reflects to find recurring failure; proposes diffs to the harness's own rules — and never lands them without a separate human apply-gate.

Stage 3Self-learning loopprereq: Stages 1–2

Goal: the project gets better at itself — Observe → Reflect → Propose → human apply-gate, over a real corpus.

region: self-learning核心问题 #3
Build
  • reusedS1 Observe: the signals already emitted — _actions.jsonl, transcripts, methodsTried.
  • newS2 Reflect: detect doom-loops, repeated failures, and scope-thrash (recurring SSOT churn).
  • newS3 Propose: staged rule / skill diffs — never auto-applied.
  • extendS4 Human apply-gate: a separate human act lands the diff (T1-style); the proposer is never the applier.
Verify — the red tests to write first

TDD gate

  1. A proposed rule diff is staged and never auto-applied — landing requires a distinct human action (producer of a rule patch ≠ authority that applies it).
  2. A synthetic doom-loop (N identical failures in the audit log) is detected and surfaced.
  3. Repeated scope-thrash across runs is flagged as its own learnable signal.
Unlocks
G3 closed: the harness compounds across runs — and, because the apply-gate is human, an ungated loop can never rewrite away its own constraints.
Section 07

What “done” means — verification discipline + coverage ledgers

This plan is executable because every stage above already carries its own definition-of-done: the red tests to write first. There is no separate “testing phase” — the test is the gate, and the gate is the spec.

The three rules that make a stage countable as done

  1. Done = a red test turned green. Each stage's TDD box is its acceptance criterion. If the test was never red first, it proves nothing.
  2. New gates are reject-before-write hooks. They plug into the existing research-op gate matrix, pull their yardstick from the Scope SSOT, and reject illegal moves at the boundary — enforce, don't instruct.
  3. The trust contracts hold throughout. At no stage does a new piece let the producer acquit its own work, edit the yardstick, or write a result into the SSOT. Any change to a base-pipeline protocol body is prepended, not rewritten.

Out of scope for v1 — deliberately deferred

To honour minimize-complexity, v1 is one project, one deep direction at a time: no multi-project portfolio; the outer multi-direction loop stays thin (the per-direction experiment loop is what is autonomous); the dashboard stays read-mostly + the Task/Objective surface, not a full GUI; self-learning proposes rules, it does not auto-tune hyperparameters. Each is a later increment, not a v1 gate.

The first concrete step

Start Stage 0 with two red tests, because they define the two seams the entire machine hangs on:

Write these first

  1. The Scope SSOT schema validates and round-trips a fixture (and rejects an out-of-schema field).
  2. An acquit op without a verdict record is REJECTED by the gate matrix.

Make them pass with the thinnest possible contract code, and Stage 0 is underway. Everything after is a deepening of a loop that, by the end of Stage 1, already runs.
