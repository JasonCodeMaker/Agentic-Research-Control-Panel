Section 01

The machine in one picture — four planes

The pipeline is four stacked planes, threaded by one backbone. The execution plane does the research; the trust plane gates what is allowed and what is true; the control plane is where the user steers; the memory plane is what compounds across runs. Above all four sits the Scope SSOT — the single home for intent every plane reads from. Each plane is the home of one concern, so nothing leaks — the single-home discipline that keeps the base pipeline trustworthy, scaled to the whole machine. It is project-agnostic: you point it at a research project and hand it a Task.

Section 02

The hierarchical Scope SSOT — the integrating backbone

Deep-learning research is non-stationary: the objective, metric, gate, architecture, even the paper's claim get rewritten mid-stream, sometimes entirely. To keep the agent aligned and the memory honest under that churn, one module sits above the four planes — a hierarchical Single Source of Truth for scope/intent, kept continuously updated, user-monitored, and read by every other component. It is not a fourth contribution; it is the backbone the three solves stand on, and it is what makes them hold when the scope moves.

Three levels — a change at one level invalidates only its dependents, and its level sets the gate strength

Project (global)
north-star objective · contribution spine · non-goals
change gate: mandatory user ratification — maximal irreversibility
Direction (regime)
hypothesis · metric · baselines · success predicate
change gate: user ratification + cross-model evidence audit
Task
active experiment · config · gate · autonomy level
change gate: agent may update + async notify (light ack)

The hierarchy is what reconciles “user-monitored” with “autonomous”: the level of a scope change sets the gate strength. A global pivot always stops for the user; a task-level metric tweak the agent can make and report. Without graduated gating, “user-monitored SSOT” collapses into supervised-mode-always and the autonomy dial (§5) is dead.

The hierarchy is a DAG, not a strict tree. A scope node — a shared baseline, dataset, metric, or even a single task — can be multi-homed under several directions at once (real trackers like Asana allow exactly this; strict single-parent nesting is a convention, not a law). This does not re-introduce a second source of truth: the node is stored once and referenced many times — multi-homing is about edges, not copies, so invariant 3 (canonical; surfaces are projections) still holds. Its one real consequence is for propagation: invalidation is reference-counted — a multi-homed node is invalidated only when its last owning parent drops or changes it; while any owner still needs it, it carries.

The non-negotiable: the SSOT owns the yardstick, never the reading

“Highest priority” must be scoped to intent. The SSOT defines what to measure and what counts as success; it must never hold or override a measured value — or it becomes the ultimate deception engine (edit the truth to fit the result). This is also how it coexists with the base pipeline's “runtime truth wins” rule: they are different layers, and a gate sits between them.

The six invariants that keep it a trust anchor, not a new failure mode

Table 1 — drop any invariant and the SSOT degrades into a deception engine, a complexity bomb, a second source of truth, or supervised-always.
InvariantWhy — the failure mode if dropped
1. Owns the yardstick, never the readingif it can assert a result, “highest priority” launders weak results — catastrophic deception
2. Versioned, not a mutable snapshot“updated” = append a new version with provenance; in-place edits lose history and enable silent goalpost-moving
3. Canonical; surfaces are projectionsplan.html / tracker / Current Best render from it (derive, don't duplicate); a second copy drifts = context pollution
4. Passive reference, not orchestratorcomponents read intent from it; it runs nothing — else a god-module bloats and fights minimize-complexity
5. Priority enforced in the gates, not declaredgates mechanically pull the yardstick/version from it, so the agent cannot act under a stale scope — enforce, don't instruct
6. Graduated gating by levelglobal = hard human gate, task = async notify; else user-monitoring kills autonomy

Scope propagation — keeping memory honest across change

Just as the base pipeline propagates landed facts to every owning surface, an SSOT update propagates the scope change to dependent memory in the same transition — every prior result, lesson, and banned idea is re-evaluated into one of three fates:

·Carryunaffected by the change — stays active
·Invalidatemeasured on an old yardstick — archived, not deleted
·Reopena failed idea may be viable under the new scope

The same transition also propagates to the autonomy dial (§5): a direction- or project-level scope change auto-reverts every affected Task toward Supervised and holds it there until the agent re-grounds under the new scope. Autonomy is earned against a shared understanding, and that understanding is purpose-relative — when the purpose shifts, the grounding the dial's freedom was granted against is stale, so prior autonomy is no longer warranted until shared context is rebuilt. This is the dial's coupling up to the SSOT: scope churn doesn't merely gate the change, it withdraws standing autonomy until alignment is restored.

Each scope transition is itself an evidence-anchored, version-stamped record (from→to, cited trigger, cause, delta, invalidates / reopens) — the causal timeline that lets the agent and the user see why the scope moved and what it cost. That transition record is the one place where moving the goalposts becomes auditable rather than silent.

Section 04

Solve #1 — the structured trust substrate

The layer that makes deception and instruction-drift structurally impossible rather than merely discouraged. It targets 核心问题 #1 — context pollution + hallucination → deception & non-compliance. Its thesis: replace trust with evidence, enforce with code not prompts. Each failure mode of problem #1 gets a mechanical counter.

Table 2 — every failure mode of problem #1, and the mechanism that forecloses it.
Failure modeCounter in the substrateStatus
Context pollution / goal drifttyped (category, status) state machine + single-home surfaces — the agent only touches what the current state legally permitsreused
Out-of-contract edits / silent non-compliancereject-before-write through one mutation surface (research-op) — illegal writes never hit disk; the agent gets a rule-violation envelope and retriesreused
Self-acquittal (judging its own work)independent cross-model judge on acquit gates — producer ≠ judge (“DRIVE but cannot ACQUIT”)new
Moving the goalposts (yardstick-gaming)the gate reads its yardstick from the Scope SSOT (§2); changing the metric is an audited, evidence-anchored SSOT transition, not a silent editnew
Hallucinated numbers / citationsmetric oracle (experiments) + citation existence check (fetch-don't-fabricate) — facts from an external authority, never weightsreused / new
Post-hoc rationalizationpre-registration (protocol commit before results) + TDD (watch the test fail first)reused
Undetectable behaviorappend-only audit log (_actions.jsonl) — every op, success or rejectreused

The substrate is deliberately thin — one mutation surface, one state machine, deterministic gates, one scope source. That is itself anti-deception: less surface for pollution, and the measured lesson that more guardrail context makes results worse (a skill dropped a task from 97% to 77% correct in the field note). This is solution direction 3 of problem #1 — minimize workflow complexity — treated as a trust mechanism, not a nicety.

The verifier, layered: DRIVE, but never ACQUIT

Because the deliverable is idea→paper, the verifier must cover both branches the survey identified: the experiment branch has an unfakeable metric oracle; the document branch has no oracle and must lean on independent critique + citation existence checks. The design answer is to layer them — a cheap deterministic gate catches mechanical cheats, freeing an expensive cross-model jury to judge meaning. All layers read their success criterion from the active Scope SSOT.

Two layers, one principle: the producer never signs off its own work

reusedL1Deterministicreject-before-write · metric oracle · cite-exists check
newL2Cross-model juryfresh, different-family judge on subjective / acquit gates
extendL3Humanfinal acquit at the dial's chosen gates (T1)

L1 is built. L3 is built (T1 acks). L2 is the single highest-leverage new piece — the survey names verifier independence the best predictor of whether a system can be trusted unsupervised, and it is exactly what the autonomy dial (Solve #2) escalates as a Task is given more freedom. Citation grounding (R2/R6) plugs into L1 as a deterministic existence check, never a model's memory.

Section 05

Solve #2 — HCI context alignment

The problem is bidirectional — the model and the user must share context both ways — so the solve is two mechanisms on one control plane, both anchored to the Scope SSOT (§2). It targets 核心问题 #2 — no reasonable HCI alignment of model↔user context.

model → user

Live research dashboard

Real-time HTML the user reads to see the model's state, claims, and evidence — and the current SSOT scope. Makes the model legible. extend reused base, widened for the autonomous / multi-task view (solution dir 1).

user → model

Task/Objective + autonomy dial

The Task/Objective surface is the task level of the SSOT; the user sets the objective and how autonomous the agent runs. new the net-new mechanism of this solve (solution dir 2).

The control plane is a management interface — human as PM, agent as managed worker

Both mechanisms above are surfaces of one stance: the control plane is a supervisory interface, not a theory of alignment. The deep-research verdict (HYBRID) found that the dominant task / project tools — Jira, Linear, Asana — converge on the same machinery: typed, gated state machines with constrained statuses, explicit transitions, admission gates, activity traces, and review points. We adopt that machinery as the GUI for human↔agent collaboration — the human is the project manager, the agent a managed worker whose tasks, states, evidence, and next actions are made visible and governable. Five primitives carry over — state machine, admission gate, objective cascade, activity log, retrospective — but the goal is swapped: not maximising completed tasks (the business-PM telos, which a self-interested worker games), but keeping the agent's work inspectable, interruptible, attributable, and verifiable.

Table 3 — the four trust properties, the PM primitives that serve each, and the gap PM alone cannot fill — which is exactly where this solve's dial and Solve #1's verifier come in.
PropertyServed by PM primitivesWhat PM does not supply
Inspectablestate machine + activity log — current state and historythe log must be a PACK'd narrative for an absent reader, not a raw trail (below)
Interruptibleadmission gate + autonomy dialthe dial is new — it comes from levels-of-automation, not from PM

The two non-empty gap cells are the project's own contributions — the dial (this solve) and the cross-model verifier (Solve #1, §4). So the interface does not replace the trust substrate; it composes with it: the PM surface makes the claim and its yardstick visible and gated, Solve #1 supplies the independent judgment at the gate, and the human's approval is captured there.

Three constraints that keep the interface honest — else PM's defaults leak back in

  1. DRIVE, but cannot ACQUIT. A human worker self-reports “done” and is trusted within known competence; an agent can hallucinate with full confidence. So strip the worker of two powers an ordinary worker has: it may not declare its own gated work successful (that is L2/L3, §4), and it may not edit the objective/yardstick. Trust comes from the verifier at the gate, never from the PM–worker relationship's implied trust — the metaphor is an interface model, not a trust-delegation model.
  2. Scope changes enter through the admission gate, not the worker's pen. Research is non-stationary, so the objective will move — but the agent proposes a revision as a Triage item and the human disposes; the objective cascade stays PM-write-only. Two primitives (gate + cascade) combine to close the goalpost-moving vector, and the move is routed through the SSOT's versioned transition (§2).
  3. The agent owes context upward — supervision ≠ alignment. The surfaces are a Common Information Space, not a store the human must mine. The agent must actively PACK evidence for an absent reader so the returning human can cheaply UNPACK it and re-ground; the activity log and dashboard are that narrative, not a compliance trail.

The fifth primitive, the retrospective, is the seam to Solve #3: its scope-revalidation face lives here (does the yardstick still hold under the churn?), while its rule-extraction face is the gated self-learning loop (§6).

The dial is the user→model instrument within that interface; the dashboard is the model→user plane shown in §1. It is set per Task, not globally — the human keeps exactly as much control as the Task's stakes warrant, and no more.

The dial is not just a UI knob — it is coupled to the trust plane: the more autonomy a Task is given, the stronger the required verifier independence and the more the loop leans on async reporting instead of human acks. The survey's lesson made mechanical — gating scales with irreversibility. (And recall §2: a direction/project-level scope change auto-reverts the dial toward Supervised until the agent re-grounds.)

What the dial buys

  • It unifies two gaps into one mechanism: the user-led Task/Objective surface (#2 solution 2) and the autonomy dial (survey W6/T7) are the same control plane — the user→model half of HCI alignment, and the task level of the SSOT.
  • It makes #1 and #2 interlock instead of trading off: choosing more autonomy automatically buys more verifier independence, so throughput and trust scale together rather than against each other.
  • The base pipeline already ships the bottom rung — T1 terminal-ack and BLOCKED-on-missing-decision are the “Supervised/Checkpoints” behaviour. The new work is the upper rungs + the per-task selector.
Section 06

Solve #3 — the self-learning loop

The outer loop that makes the project get better at itself. It targets 核心问题 #3 — no personalized project self-learning. It reads the signals the rest of the machine emits, distils durable lessons, and proposes changes to the harness's own rules — but never lands them without a human apply-gate.

Observe → reflect → propose → human-gated land

reusedS1Observeaudit log + transcripts + methodsTried
newS2Reflectdetect doom-loops, repeated failures, scope thrash
newS3Proposerule / skill diffs, staged — never auto-applied
extendS4Human apply-gatea separate human act lands the diff (T1-style)

Two memory tiers feed it: the reused content memory (learnings, methodsTried, the failed-idea banlist that privileges negative results) and the new harness self-learning (lessons that rewrite the system's own rules). Every memory entry is stamped with the Scope SSOT version that produced it (§2), so on a scope transition the carry / invalidate / reopen pass runs and the banlist is scope-conditional, not absolute — recurring scope thrash is itself a doom-loop the reflector surfaces. The discipline is the survey's and the field note's — fix the harness, not the output, and the producer of a rule patch is never the authority that applies it, because an ungated loop can rewrite away its own constraints. It is built last because it consumes exactly what the other solves emit.

What this picture hands to the build-order plan

  1. The build surface is five net-new regions on a working substrate: the Scope SSOT backbone, the upstream roles, L2 cross-model jury + citation gate, the control plane / autonomy dial, and the self-learning loop.
  2. The Stage 1 walking skeleton now has a concrete shape: one Task on the SSOT → thin Scope/Search/Ideate stubs → the reused Experiment loop → thin Writer → through the real gates (reading the SSOT yardstick) → rendered to the dashboard, run at one chosen autonomy level.
  3. The three solves are co-equal, but their build leverage differs: stand up the SSOT backbone early (everything reads from it), deepen L2 first (the trust ceiling and what the dial escalates), then the upstream roles; add the self-learning loop last, since it consumes what the others emit.
